## §1. Контекст и цель

### §1.1. Зачем

Перед брифингом #8 (летняя смена тарифа) нужно починить UX форм учеников и семей. Иначе на «летний скрипт» пойдут битые данные.

### §1.2. Прецедент

Ученик `3f552f6e` Дмитрий Петров создан с `start_date = end_date = 24.04.2026` при `status = "Активен"`. Поля `end_date` и `status` противоречат друг другу. При создании поля `end_date` в форме нет, но значение всё равно попадает — это и есть Задача 0, требующая диагностики.

### §1.3. Что меняем (резюме)

1. Диагностика и устранение автозаполнения `end_date` при создании.
2. Форма «Новый ученик»: добавить `start_date`, убрать `статус` и `end_date`.
3. Кнопка «Выбытие» на Detail view ученика (одной транзакцией: `end_date` + `status="Выбыл"`).
4. Кнопка «Добавить ребёнка» на Detail view семьи.

### §1.4. Чего НЕ меняем

- Бот `bot_close_period` (включая Branch, ForEach, тексты сообщений) — **не трогать**.
- Структуру дашборда — **не трогать**.
- Состав и порядок колонок в таблицах `students`, `families`, `student_tariffs`, `tariffs`, `charges` — **не трогать**.

## §4. Задачи

### §4.0. Диагностика: автозаполнение `end_date`

**Симптом.** При создании нового ученика в `end_date` попадает дата создания. В форме «Новый ученик» поля `end_date` нет, но значение всё равно заполняется.

**Где искать** (Data → Columns → таблица `students` → колонка `end_date`):

| # | Поле | Что проверить |
|---|------|---------------|
| 4.0.1 | `Initial Value` | Если `TODAY()`, `NOW()`, `[start_date]` или подобное — это источник. **Очистить полностью.** |
| 4.0.2 | `App Formula` | Не должно быть. Если есть — очистить. |
| 4.0.3 | `Suggested Values` | Маловероятно, но проверить. |
| 4.0.4 | `Reset on edit?` | Если стоит — снять. |

**Результат:** `end_date` пуст при создании, заполняется только через действие «Выбытие» (§4.2).

**В отчёт:** что именно вызывало автозаполнение, что было удалено.

### §4.1. Форма «Новый ученик»

**Текущее состояние** (см. скрин в чате 06.05): форма содержит имя, фамилия, семья, класс, статус (Активен/Выбыл).

**Цель:**

| # | Поле | Действие |
|---|------|----------|
| 4.1.1 | `start_date` | Добавить в форму. Тип Date. Выбор из календаря. `Require?` = `TRUE`. |
| 4.1.2 | `status` | Убрать из формы новой записи. `Initial Value` = `"Активен"`. В форме редактирования — оставить видимым (нужно для Задачи 2 fallback). |
| 4.1.3 | `end_date` | Убрать из формы новой записи. `Initial Value` — пустой (после §4.0). В форме редактирования — оставить видимым (но read-only, изменяется только через «Выбытие»). |

**Способ реализации — на выбор Sonnet'а** (выбрать один, описать в отчёте):

**Вариант А — через `Show?` с условием:**
- Для `status` и `end_date` поставить `Show?` = `NOT(ISBLANK([_THISROW].[student_id]))` — показывать только в режиме редактирования.

**Вариант Б — через отдельный Slice:**
- Создать Slice `students_create` с урезанным набором колонок (без `status`, `end_date`).
- Привязать к нему отдельный form view для создания.

Если упрёшься в неожиданное поведение AppSheet (например, Show_If не работает на form view) — переключиться на другой вариант, не пытаться чинить.

**В отчёт:** какой вариант выбран и почему.

### §4.2. Кнопка «Выбытие»

**Где:** Detail view ученика (existing).

**Поведение (одна транзакция):**
- Пользователь нажимает «Выбытие» → выбирает дату из календаря → одним действием:
  - `end_date` = выбранная дата,
  - `status` = `"Выбыл"`.

**Реализация (вариант А — основной):**

| # | Параметр Action | Значение |
|---|-----------------|----------|
| 4.2.1 | Action name | `students__withdrawal` |
| 4.2.2 | For a record of this table | `students` |
| 4.2.3 | Do this | `Data: set the values of some columns in this row` |
| 4.2.4 | Inputs | `withdrawal_date` тип Date, `Required` = `TRUE` |
| 4.2.5 | Set columns | `end_date` = `[withdrawal_date]`, `status` = `"Выбыл"` |
| 4.2.6 | Display name | `Выбытие` |
| 4.2.7 | Show_If | `AND([status] = "Активен", LOOKUP(USEREMAIL(), "users", "email", "role") = "ADMIN")` |
| 4.2.8 | Position | `Prominently` |
| 4.2.9 | Confirmation | OFF (выбор даты уже подтверждение) |

**Если вариант А упрётся** в ограничение AppSheet (например, Inputs не работают для `set values`) — fallback вариант Б:

| # | Параметр | Значение |
|---|----------|----------|
| 4.2.Б | Composite action | Action 1 (set `end_date` через input) → Action 2 (set `status = "Выбыл"`) |

**В отчёт:** какой вариант сработал, какие ограничения встретились.

**Доступность по периоду:** действие работает в любой момент, без зависимости от текущего периода. Текущее начисление за уже открытый период **не пересчитывается** (см. §5.2).

## §5. Архитектурный контекст (справочно, НЕ ПРАВИТЬ)

Эти моменты — для понимания. **Менять не нужно.**

### §5.1. Текст NO-сообщения

В `step_no_set_result` уже переписан 06.05 вечером:

```
CONCATENATE("Период ", ANY(settings[current_period]), " закрыт. Открытый период: ",
IFS(
  NUMBER(RIGHT(ANY(settings[current_period]), 2)) < 12,
  NUMBER(LEFT(ANY(settings[current_period]), 4)) * 100 + NUMBER(RIGHT(ANY(settings[current_period]), 2)) + 1,
  TRUE,
  (NUMBER(LEFT(ANY(settings[current_period]), 4)) + 1) * 100 + 1
),
". Создано начислений: ", COUNT(SELECT(charges[charge_id], [period] = ANY(settings[current_period]))), ".")
```

«Должников» убрано из-за расхождения pre/post-commit (§7.2 v10). **Не трогать.**

### §5.2. Логика выбытия и начислений

При выбытии в открытом периоде текущее начисление **не пересчитывается** (начисления не дробим). При следующем закрытии новых начислений выбывшему ученику не будет — это работает автоматически через существующий фильтр `[status] = "Активен"` в `step_no_create_charges` (§5.7 v10). Никаких правок в боте `bot_close_period` не требуется.

### §5.3. Статус полей `start_date` / `end_date`

В v10 §8.2 эти поля числились как «рудименты к удалению». Эта классификация **отменяется**: оба поля теперь рабочие. В v11 backup — будет обновлено.

### §5.4. Две оси `tariff_type` vs `charge_type`

См. §2.3 v10. **Не путать в формулах.** В этом брифинге формул с этими полями нет, но если по ходу столкнёшься — свериться с §2.3 v10.
