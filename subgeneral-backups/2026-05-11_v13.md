## §1. Что произошло за #13 (08–11.05.2026)

Ветка #13 началась 08.05.2026 с реструктуризации корневых файлов (v12.2) и закрылась 11.05.2026 после серии из четырёх pre-патчей. Главный архитектурный итог — **защита «закрытый период = read-only»** на уровне UI для всех трёх таблиц с временной осью (charges, payments, expenses), включая Edit и Delete. Параллельно зачищены два рудимента (`tariff_type`, `billing_frequency`).

### §1.1. pre-#08.3 (10.05.2026) — UX-патч заголовков форм

Цель: дать формам понятные заголовки при создании и редактировании. **Технически невозможно** в AppSheet (май 2026) — раздельные Add/Edit form views на одной таблице не разделяются (SKILL Q.17). Принят компромисс — VC `Section_Header` с универсальным заголовком «Создать/редактировать …» на единственной системной form view (SKILL Q.16). Затронуто 6 сущностей.

Два инцидента дисциплины SubGeneral в этой сессии (Инциденты 7-8 в `REGULATIONS.md` §1.7): бриф составлялся без `grep`'а SKILL'а, после Инцидента 7 grep сделан по неполным ключам — решение по VC лежало в строке 122 SKILL'а с момента pre-#06. Регламентное следствие: `REGULATIONS.md` §7 «Дисциплина SubGeneral».

Внебрифингово в той же сессии: `tariff_type` убран из `tariffs_Detail`, `students[birth_date]` Initial value очищен (обе правки от владельца).

### §1.2. pre-#08.4 (11.05.2026) — Удаление `tariff_type`

Цель: убрать колонку-рудимент `tariffs[tariff_type]` (значение «Единоразовый» в проекте никогда не использовалось, все тарифы регулярные). Фильтр в `bot_close_period` упрощён до `[student_id].[status] = "Активен"`. При Regenerate Schema всплыла дополнительная ссылка в VC `settings[current_period]` (фильтр `[tariff_id].[tariff_type] = "Регулярный"`) — снята Sonnet'ом в той же сессии, формула стала `INDEX(SORT(SELECT(charges[period], TRUE), TRUE), 1)`.

Внебрифинговый ресёрч встроенного фильтра AppSheet: на временном table view `начисления` Filter panel предложила только `описание` и `создан`, не предложила ни `period` (Ref), ни `student_id` (Ref), ни `amount` (Number), ни `charge_type` (Enum). Гипотеза «фильтрует только скаляры» не подтверждена. Правило не установлено — оставлено как открытое исследование в `SYSTEM.md` §8.4.

### §1.3. pre-#08.5 (11.05.2026) — UX-чистка + защита твёрдой земли (этап 1)

Пять задач:
- §4.1 `billing_frequency` — Show? = false (удаление столбца отложено в pre-#08.6).
- §4.2 форма ученика — `students_visible_Form` очищена от VC задолженностей и Related-блоков.
- §4.3 inline платежи — закрыт по факту, table view в карточке ученика уже отображает «тип / сумма / дата / комментарий».
- §4.4 **запрет Edit для charges из закрытых периодов** — Show_If `[period] = INDEX(settings[current_period], 1)` на системном Edit. Изначально планировалось через `charge_type = "Авто"`, в диалоге 11.05 переформулировано: твёрдая земля = весь закрытый период, независимо от типа charge. Системный Edit в отличие от системного Add (pre-#08.3 SKILL Q.17) **принимает** Show_If — read-only ограничение касалось только Add.
- §4.5 фиксация источника `charge_type` — Initial value «Ручное» (уже стояло до сессии), поле убрано из `charges_Form`. Бот через `student_tariffs__create_next_charge` явно пишет «Авто».

Архитектурные фиксации:
- Паттерн проекта `INDEX(settings[<col>], 1)`, не `ANY(...)` — подтверждён владельцем, записан в auto-memory.
- Двухуровневый канал правки данных (`SYSTEM.md` §7.1): AppSheet UI блокирует закрытое для повседневной работы, прямое редактирование Google Sheets — аварийный канал владельца (новая роль «Администратор Google Sheets» в `GLOSSARY.md` §4).

### §1.4. pre-#08.6 (11.05.2026) — Защита твёрдой земли (этап 2) + закрытие рудиментов

Две задачи по брифу + плановое расширение в ходе сессии:

**По брифу:**
- §4.1 `billing_frequency` удалён из листа `tariffs` (Google Sheets, столбец D), Regenerate Schema без ошибок.
- §4.2 защита от **создания** charges в закрытом периоде: на `charges[period]` Initial value `INDEX(settings[current_period], 1)` и Valid_if `SELECT(ref_periods[period], AND(NUMBER([period]) >= NUMBER(INDEX(settings[current_period], 1)), NUMBER([period]) <= NUMBER(INDEX(settings[current_period], 1)) + 3))` — окно current..current+3. Окно ±3 — паттерн проекта (SKILL §547). `ref_periods` переведена в Read-Only, исчезла кнопка «+ New» в Ref-dropdown.

**Плановое расширение по решению стратега:** распространение защиты «закрытый период = read-only» на Edit + Delete для всех трёх таблиц:
- `charges`: Delete → Show_If `[period] = INDEX(settings[current_period], 1)`.
- `payments`: Edit + Delete → Show_If `TEXT([payment_date], "YYYYMM") = INDEX(settings[current_period], 1)` (у платежей нет колонки period, вычисляется из payment_date — архитектурно правильно).
- `expenses`: Edit + Delete → Show_If `[period] = INDEX(settings[current_period], 1)`.

**Открытие в ходе сессии:** **slice с Update mode = Read-Only блокирует системные actions в slice-Detail**, даже при корректных Show_If. Три slice'а (`charges_filtered`, `payments_filtered`, `expenses_filtered`) переведены в Updates+Deletes. Паттерн зафиксирован в SKILL R.2.

Попутно исправлен баг `expenses[period]` Valid_if (была `- 3` на нижней границе, разрешала ретро).

**Открытый плавающий дефект:** Delete на мобильном через slice-Detail для `charges` и `expenses` не виден, через `payments` — виден. Полный logout/login не помог. Уточнённый паттерн — зависит от **точки входа**: через filter-view → Delete отсутствует; через карточку ученика / меню «расходы» → Delete виден. Десктоп корректен во всех маршрутах. Не блокер: показ Show_If сохраняется в любом случае, твёрдая земля не нарушена. Зафиксировано в SKILL R.3 как floating defect.
