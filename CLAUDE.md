Операционная инструкция для Claude. Читается автоматически в начале каждой сессии Claude Code в этом репозитории.

## 2. Структура репозитория

```

├── GLOSSARY.md              ← словарь, метафоры

├── REGULATIONS.md           ← регламенты работы (todo)
├── RECOVERY.md              ← Disaster Recovery (todo, в #09)
├── INDEX.md                 ← реестр всех бэкапов/брифингов/отчётов
│
├── briefings/               ← задания Sonnet'у (append-only)
├── reports/                 ← отчёты Sonnet (append-only)
├── subgeneral-backups/      ← состояния проекта по датам
```

`briefings/` и `reports/` — **никогда не редактируются** после создания. История.

## 3. Старт сессии — обязательный чек-лист

В таком порядке:

1. Прочитать `SYSTEM.md` (бизнес-модель), `GLOSSARY.md` (метафоры и термины), `INDEX.md` (реестр артефактов), последний бэкап в `subgeneral-backups/` (текущее рабочее состояние).
2. По `INDEX.md` проверить дельту: если после даты последнего бэкапа есть новые отчёты/брифинги — прочитать их, чтобы знать, что произошло между бэкапом и «сейчас».
3. `README.md` — только если нужна навигация по репозиторию.
4. Спросить у владельца:
   - Сегодняшнюю физическую дату.
   - Текущую версию AppSheet.
   - Приоритет на день.
   - Изменения после последнего бэкапа.

Без этого чек-листа — не начинать содержательную работу.

## 4. Стиль общения

- **Короткие ответы.** 3 предложения вместо абзаца. 2 вместо 3.
- **В длинных сессиях (UI-проводка по AppSheet, развёртывания) — особенно коротко.** Длинные расплывчатые сообщения в таком режиме изматывают владельца. Одно действие на сообщение. Одна-две строки. Таблицы и многоуровневые планы — только когда явно попросили или критично для решения.
- **Один вопрос за раз.** Не задавать сразу пять.
- **Метафоры — рабочий язык.** «Сосуды», «твёрдая земля», «живая зона», «суровая правда без шума» — терминология проекта, не украшения. См. `GLOSSARY.md`.
- **Сигнал перегруза:** если владелец говорит «много написал», «не понял», «можно сжать» — сократить в 2-3 раза. Не переписывать другими словами — выкидывать половину пунктов или идти последовательно.
- **При указании на ошибку** — признать и исправить (или показать, в чём владелец сам неправ). **Не оправдываться** — оправдания это шум.

## 5. Дисциплина (для брифингов Sonnet'у и для собственной работы)

1. **Никаких диагностических формул в живых Sheets** — даже временных.
2. **Видишь смежное полезное действие за пределами задания — фиксируй как хвост, не делай.** (Прецедент #07.3.)
3. **В отчётах разделять чётко:** что сделал Sonnet, что — владелец.
4. **Категоричные утверждения о механиках AppSheet — проверять.** Первый источник — `sonnet-skills/appsheet-setup-SKILL.md` (накопленный опыт Sonnet'а по проекту), затем официальная документация AppSheet.
5. **Архитектурные правки — только через стратега.** Не согласовывать напрямую с владельцем минуя SubGeneral.
6. **Имена Enum-значений сверять с проектом** перед написанием формул.
7. **Индекс — атомарно с артефактом.** Любое создание/перенос файла в `subgeneral-backups/`, `briefings/`, `reports/` сопровождается строкой в `INDEX.md` в том же ходе. Файл создан и не отражён в индексе = работа не закончена. Без этого преемник снова будет «реконструировать по reports/», как 09.05.2026.

Перед любыми правками — пара бэкапов:

Без подтверждения бэкапа от владельца — **не давать команду Sonnet'у начинать работу.**

## 7. Работа в Claude Code

**Файлы базы знаний** — доступны напрямую через Read/Write/Edit. Не пересказывать в чате — править файл.

**Брифинги и отчёты:**
- Брифинги — пишу сам в `briefings/YYYY-MM-DD_briefing-XXX.md`, без черновика в чате (договорённость 08.05.2026).
- Отчёты от Sonnet — кладу в `reports/`, как только владелец передаст текст.

**Git:**
- Коммиты и push **делает владелец сам** (для контроля изменений перед фиксацией). Не запускать `git add/commit/push`.
- После правок — сообщить владельцу что сделано, остановиться. Можно показать `git status` или `git diff` для удобства.
- Рекомендуемый стиль сообщений коммита (на случай, если владелец спросит): на английском, кратко по делу, в повелительном наклонении — `Add briefing pre-#08.1`, `Update SYSTEM.md §4.4`.

**Adaptive thinking** — включать только при:
- Отладке гипотез (как каскадирование §6.1 v10).
- Сложных архитектурных решениях с прогонкой во времени.
- Когда нужно реально подумать «во времени» — что произойдёт ДО, что ПОСЛЕ транзакции.

**Не включать для:** составления брифингов, обновления документации, простого диалога. Лимиты Pro расходуются.

## 8. Закрытие ветки

Перед завершением сессии:
1. Создать новый файл в `subgeneral-backups/YYYY-MM-DD_v<N>.md` (короткий формат — только текущее состояние и хвосты, без дублирования с SYSTEM/GLOSSARY).
2. **Сразу добавить строку в `INDEX.md` §1** — дата, файл, версия AppSheet, тип, суть. Обновить «Текущий рабочий бэкап». См. §5.7.
3. Если изменилась бизнес-модель — обновить `SYSTEM.md`, версию документа.
4. Если изменился словарь — обновить `GLOSSARY.md`.
5. Закоммитить и запушить (это делает владелец).
