## §1. Контекст и цель

### §1.1. Зачем

Pre-#08 закрыл базовые UX-проблемы (форма создания, выбытие как одно действие, добавление ребёнка). В ходе ревизии вылезли дополнительные:

1. **Дата рождения отсутствует** в карточке ученика — критично для школы.
2. **Выбытие неполное** — нужно очищать `class_group` и удалять `student_tariffs`.
3. **Должен быть фильтр видимости списка учеников** — выбывшие с погашенными долгами засоряют список.
4. **Технические `*_id` поля видны в карточках** — мусор для пользователя.
5. **`created_at` в карточках по-разному называется** — единый стиль нужен.
6. **На view-фильтрах (начисления/платежи/расходы) лишние Edit/Delete/Add** — могут привести к случайным изменениям.
7. **В карточке расхода нет Delete** — невозможно удалить ошибочно внесённый расход.

**Связь с хвостами из отчёта pre-#08 §9:**
- Хвост №1 (`_ComputedName` для выбывших — класс в имени) — закрывается косвенно через §4.1.2 (`class_group = ""` при выбытии). Если формула `_ComputedName` опирается на `[class_group]` — проблема снимется автоматически. Проверить в тестах §7.6.
- Хвост №2 (деактивация `student_tariffs` при выбытии) — полностью закрывается §4.1.3 (удаление строк).
- Хвост №3 (ручной выбор даты выбытия вместо `TODAY()`) — полностью закрывается §4.1.1.

### §1.2. Что меняем (резюме)

- §4.1. Расширение action «Выбытие»: + очистка `class_group` + удаление `student_tariffs` + Confirmation.
- §4.2. Новое поле `students[birth_date]`.
- §4.3. Slice `students_visible` (фильтр видимости списка).
- §4.4. Унификация полей в Detail view: скрыть `*_id`, переименовать `created_at` → «создан», для расходов — переименовать `period`, добавить Delete.
- §4.5. Чистка view-фильтров: убрать Edit/Delete/Add из самих фильтров.

### §1.3. Чего НЕ меняем

- Бот `bot_close_period` (Branch, ForEach, тексты сообщений) — не трогать.
- Структуру дашборда — не трогать.
- View-фильтры (`фильтр начислений`, `фильтр платежей`, `фильтр расходов`) **не удаляем.** Только чистка лишних кнопок. Удаление — отдельным мини-патчем после периода тестирования встроенного фильтра AppSheet.

## §4. Задачи

### §4.1. Расширение action «Выбытие»

**Контекст:** в pre-#08 создан action `students__withdrawal` (одна транзакция: `end_date = TODAY()` + `status="Выбыл"` + Confirmation). Состояние на входе в pre-#08.1 — см. отчёт pre-#08 §5. Расширяем по трём осям: ручной выбор даты выбытия, очистка `class_group`, удаление `student_tariffs`.

**Что добавить к существующему action:**

| # | Что | Зачем |
|---|-----|-------|
| 4.1.1 | Ручной выбор `end_date` пользователем (вместо `TODAY()`) | Дата выбытия может отличаться от дня нажатия кнопки |
| 4.1.2 | `class_group` = пусто | Выбывший не должен маячить в группах класса |
| 4.1.3 | Удалить все строки `student_tariffs` где `[student_id] = THISROW.[student_id]` | В карточке выбывшего не должны висеть привязки тарифов. Историю не сохраняем (Р-14, БД компактнее) |

**Confirmation уже существует** (добавлен в pre-#08, §5 отчёта). Переиспользуем как есть, **не дублировать**:
```
CONCATENATE("Вы уверены, что ученик ", [first_name], " ", [last_name], " заканчивает обучение в периоде ", ANY(settings[current_period]), "?")
```

**Целевая последовательность (UX вариант А):**
1. Пользователь жмёт «Выбытие».
2. Видит Confirmation → Да/Нет.
3. Если Да — открывается форма с одним полем (выбор даты выбытия).
4. Сохраняет → транзакция применяется (`end_date`, `status`, `class_group`, удаление `student_tariffs`).

---

**Реализация — два пути, в порядке приоритета.**

**Путь 1 (предпочтительный) — Action with input.**

Если ваша версия AppSheet поддерживает (относительно свежая фича — проверить наличие в UI: Behavior → Actions → у `Data: set the values of some columns in this row` появляется секция «Inputs»):

- В `students__withdrawal` добавить **Input** с именем `withdrawal_date`, тип `Date`, обязательное.
- В Set columns: `end_date = [withdrawal_date]` (вместо `TODAY()`), `class_group = ""`.
- Confirmation остаётся.
- **Что произойдёт:** после подтверждения Confirmation AppSheet сам откроет диалог ввода даты, потом применит транзакцию. Без новых колонок и Slice.

**Путь 2 (fallback, если Inputs недоступны) — Slice + Form view + LINKTOFORM.**

- Добавить колонку `students[withdrawal_date]` тип Date в Sheets, Regenerate Schema.
  - `Show?` = `FALSE` в Detail view (служебная, не показываем как обычное поле).
  - `Require?` = `FALSE` в общей форме создания.
- Создать Slice `students_withdrawal` (source: `students`, all columns, Updates only).
- Создать Form view `students_withdrawal_form` на этом Slice, Column order: только `withdrawal_date`.
- Изменить `students__withdrawal`: тип сменить на `App: go to another view within this app`, Target: `LINKTOFORM("students_withdrawal_form", "withdrawal_date", TODAY())` (TODAY как дефолт в форме, пользователь меняет).
- Confirmation — оставить на этом же action (срабатывает до открытия формы).
- На Form Saved этого view — повесить отдельный action `students__withdrawal_apply` типа `Data: set the values of some columns in this row`:
  - `end_date = [withdrawal_date]`,
  - `status = "Выбыл"`,
  - `class_group = ""`.

**Удаление `student_tariffs` (общий шаг для обоих путей):**

- Создать вспомогательный action `student_tariffs__delete_one` (`Data: delete this row`, table: `student_tariffs`).
- Добавить в `students__withdrawal` (Путь 1) или `students__withdrawal_apply` (Путь 2) шаг **«Execute an action on a set of rows»**:
  - Referenced table: `student_tariffs`.
  - Referenced rows: `SELECT(student_tariffs[assignment_id], [student_id] = [_THISROW].[student_id])`.
  - Referenced action: `student_tariffs__delete_one`.
- Если для последовательности шагов нужен **Grouped action** — обернуть, Confirmation вешать на Grouped в целом (только для Пути 1; в Пути 2 Confirmation на исходном action, цепочка через Form Saved).

**В отчёт:** какой путь сработал (1 или 2), какие ограничения встретились, итоговая структура actions.

- **Form views** — НЕ трогать.

### §4.4. Унификация полей в Detail view

Это **общие правила**, применимые ко всем Detail view проекта. Пройдись по каждому Detail view и применить.

**Правило 1: скрыть KEY-колонки.**
- В каждом Detail view скрыть колонки вида `*_id` (charge_id, expense_id, payment_id, и т.д.), `_ComputedKey`.
- **Способ:** Data → Columns → колонка → `Show?` = `FALSE`. Если AppSheet не даёт скрыть на writable-таблице (BP-30/35) — оставить, но через Column order Manual вынести в конец списка.
- Где применимо: charges, expenses, payments, students, families, student_tariffs, tariffs.

**Правило 2: переименовать `created_at` в `"создан"`.**
- В каждой таблице, где есть `created_at` — поставить Display name = `"создан"`.
- Где: charges, expenses, payments (и любые другие, если есть).

**Специально для расходов:**
- Поле `period` — переименовать в `"период"`.
- В Detail view расхода **добавить Delete** (Edit там уже есть, нужен ещё Delete для исправления ошибочных записей). Системный action — через UX → Detail view → System actions.

**Специально для платежей (карточка одного платежа):**
- Скрыть `_ComputedKey`.
- Скрыть `class_group` (он дублирует информацию из `ученик`).
- Оставить ссылку на ученика, дату, сумму, тип, комментарий, переименованный «создан».

**В отчёт:** список всех таблиц и что в каждой переименовано/скрыто.

### §4.5. Чистка view-фильтров

**Контекст:** view-фильтры (`фильтр начислений`, `фильтр платежей`, `фильтр расходов`) — это компромиссное решение, которое было реализовано когда не знали про встроенный фильтр AppSheet. Сейчас встроенный фильтр работает на основных view (`расходы`, `платежи` из карточки ученика, и т.д.), и эти фильтры станут рудиментами.

**В этом брифинге их не удаляем** — нужен период тестирования встроенного фильтра. Только убираем мусор.

**Что чистим:**

| View | Что убрать |
|------|-----------|
| `фильтр начислений` (верхний блок Detail view) | Edit, корзина (Delete) |
| `начисления фильтр список` (нижний ref-list) | Add (новые charges не создаём через фильтр) |
| `фильтр платежей` (верхний блок) | Edit, корзина |
| `платежи фильтр список` (нижний) | Add (платежи добавляются через карточку ученика) |
| `фильтр расходов` (верхний блок) | Edit, корзина |
| `расходы фильтр список` (нижний) | Add (расходы добавляются через основной view «расходы») |

**Способ:** в каждом Detail view → раздел System actions / Behavior → отключить лишние действия. Или через `Show_If = FALSE` на конкретных system actions.

**В отчёт:** скрин каждого почищенного фильтра.

## §5. Архитектурный контекст (справочно, НЕ ПРАВИТЬ)

### §5.1. Действие «Выбытие» — почему расширяем

Карточка выбывшего ученика остаётся в системе для истории и долгов. Но визуально она должна быть **чистой**: без класса (он уже не в этом классе), без тарифов (он уже не учится). История привязок тарифов в проекте не хранится (Р-14), поэтому удаление `student_tariffs` — без потерь.

### §5.2. Логика выбытия и начислений

При выбытии в открытом периоде текущее начисление **не пересчитывается** (начисления не дробим). При следующем закрытии — новых начислений выбывшему не будет (фильтр `[status] = "Активен"` в `step_no_create_charges`). Никаких правок в боте `bot_close_period` не нужно.

### §5.3. Архитектурный приоритет — простота

Принцип «суровая правда без шума» (см. `GLOSSARY.md` §1.3): не закладываем механики, которые ничего не дают сейчас. Удалить `student_tariffs` при выбытии — простая операция. Альтернатива «хранить историю с датами» — отвергнута давно (Р-14).
