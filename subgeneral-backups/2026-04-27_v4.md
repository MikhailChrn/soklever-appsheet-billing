- `filter_date_to` (Date) — устарела

- `общий_долг` — `SUM(SELECT(families[family_debt], [family_debt] > 0))`
- `поступления_год`, `расходы_год` — суммы с `academic_year_start`

### 2.5 Расхождения с моделью v3, выявленные при работе с AppSheet

При реализации Sonnet выяснилось, что фактические имена колонок в Sheets отличаются от описания в v3:

- `students[class]` фактически называется **`class_group`**
- `students[student_name]` фактически разнесено на **`first_name` + `last_name`**

В этом бэкапе (v4) использованы фактические имена. При обновлении xlsx до v7 — синхронизировать.

## 4. Метафоры и общий язык

### 4.1 Метафора «двух сообщающихся сосудов»

> **Начисления — один сосуд, платежи — другой. Уровни должны быть равны.**
> Перекос в сторону начислений → долг (красный флаг).
> Перекос в сторону платежей → переплата (аванс).
> `opening_balances` = стартовая отметка «сколько уже было налито в каждый сосуд до того, как мы начали смотреть».
> Привязки конкретных капель платежа к каплям начисления нет — только общий уровень.

**Следствия:**
- `debt = opening_balance + SUM(charges) - SUM(payments)` за всё время
- Платёж не «гасит конкретное начисление» — он просто меняет уровень
- Частичная оплата, оплата задним числом, аванс — обрабатываются одинаково
- Нельзя ответить на вопрос «за какой именно месяц долг» (и не нужно)

**Используется при:** объяснении модели супруге-оператору, обсуждении «зачем не привязываем платёж к charge», обсуждении переплат и опозданий.

### 4.2 Параллельные системы

> **Sheets — для владельца. AppSheet — для оператора. Разные инструменты для разных людей.**

**Следствие:** лист `dashboard` в Sheets к AppSheet не подключаем — это владельческий инструмент. Все операторские сводки и фильтры — штатными средствами AppSheet поверх таблиц.

## 5. Принятые архитектурные решения за ветку

### 5.1 Дашборд — вариант A
Реализован штатными средствами AppSheet поверх таблиц. Лист `dashboard` в Sheets к AppSheet не подключён. Fallback при упоре в вёрстку — Settings table внутри AppSheet (использован).

### 5.2 Settings table как паттерн
Служебная таблица `settings` (одна строка) — основа для:
- Цифровых блоков дашборда (через virtual columns)
- Параметров фильтров (фактические колонки)
- Хранения «текущего периода» через vc

### 5.3 Категории расходов — нормализация
Вынесены в отдельную таблицу-справочник `expense_categories`. `expenses[category]` и `settings[filter_category]` — Ref на неё. Технический долг по денормализации закрыт сразу, не накопился.

### 5.4 Период — Enum/Text, не Number
Не нормализуется в отдельную таблицу. `period` — это вычисляемая характеристика времени (формат YYYYMM), а не справочник. Сравнение через `NUMBER([period])` для арифметических операций.

### 5.5 Платежи и начисления — модель «двух сосудов»
Платёж не привязывается к конкретному начислению. `debt` считается как разность накопительных сумм. См. метафору 4.1.

### 5.6 UX-разделение быстрая регистрация / аналитика
Primary — для быстрой регистрации (сквозной список + кнопка «+»). Menu — для аналитики (Dashboard с фильтрами).

### 5.7 Карточка ученика — компактность
В карточке только то, что нужно оператору при звонке родителя: имя, класс, текущий долг, последние 150 дней платежей и начислений. Полная история — отдельным путём.

## 6. Best practices проекта

Накопленный практический опыт. Обязательны для соблюдения, не пересматривать без согласования.

### Архитектурные

**BP-1.** Sheets — источник истины для владельца. AppSheet — операторский UI. Лист `dashboard` к AppSheet не подключаем.
**BP-2.** `settings` — служебная таблица для текущих параметров (одна строка). Поля фильтров, активный период, virtual columns дашборда живут здесь. Не свалка.
**BP-3.** `expense_categories` — справочник, Read-Only для оператора.
**BP-4.** Нормализация vs Enum: нет атрибутов и детерминирован → Enum. Есть именование и рост → таблица-справочник + Ref.

### Формульные паттерны

**BP-5.** Чтение `settings`: `ANY(settings[column])` или `INDEX(settings[column], 1)`.
**BP-6.** Фильтр Slice «пусто = все»: `OR(ISBLANK(ANY(settings[field])), [field] = ANY(settings[field]))`.
**BP-7.** **MAX не работает на Enum/Text-колонках.** Использовать `INDEX(SORT(SELECT(...), TRUE), 1)`.
**BP-8.** **TEXT()** принимает только Date, не Number.
**BP-9.** Display name view — внутри `INDEX(table[column], 1)` или `ANY(table[column])`.
**BP-10.** Virtual columns в Test-режиме возвращают 0 — нормально.
**BP-11.** «Текущий период» = `INDEX(SORT(SELECT(charges[period], [tariff_id].[tariff_type] = "Регулярный"), TRUE), 1)`.

### UX и поведение

**BP-12.** Кнопки «+» появляются автоматически в view с Adds-разрешённым источником. Отдельные Action для добавления — не плодим.
**BP-13.** Action в Dashboard view иногда не срабатывает — известное ограничение. Fallback через навигацию или view без Dashboard-обёртки.
**BP-14.** Initial value для предзаполнения форм: `IFS(NOT(ISBLANK(ANY(settings[filter_field]))), ANY(settings[filter_field]))`.

### Типы и Ref

**BP-15.** FK через Ref, не через текстовые сравнения. Сравнение `[a] = [b]` напрямую, без TEXT-обёрток. Если ругается — проверять типы.
**BP-16.** LABEL целевой таблицы — осмысленная колонка (имя, название), не KEY.

### Слайсы

**BP-17.** Slice пересоздавать, если формула не сохраняется (баг кеширования AppSheet).
**BP-18.** Update mode Slice = `ADDS_AND_UPDATES`, чтобы кнопка «+» появилась.

### Формы и заголовки (добавлено 27.04.2026)

**BP-19. Заголовок формы создания через виртуальную колонку Show.**
- Type: `Show`, Category: `Section_Header`
- App formula: `""`
- Display name: нужный текст
- Show?: `true` (или условие, отличающее режим создания)
- В Column order формы — первой
- При Manual order добавлять только колонки своей таблицы.

**BP-20. Сравнение Enum-периода (YYYYMM) с динамической датой.**
```
NUMBER([period]) >= (YEAR(TODAY() - N) * 100 + MONTH(TODAY() - N))
```
Формат YYYYMM при числовом сравнении даёт правильный хронологический порядок.

**BP-21. CONCATENATE числовых значений без TEXT().**
AppSheet автоматически приводит числа к строке внутри CONCATENATE:
```
CONCATENATE(YEAR(date), IF(MONTH(date) < 10, "0", ""), MONTH(date))
```

**BP-22. NUMBER() вместо VALUE().**
Функции `VALUE()` в AppSheet нет. Для парсинга Text/Enum в число — `NUMBER()`.

## 7. Стандарты работы мультиагентной системы

В этой ветке сложилась устойчивая схема Owner / Opus / Sonnet. Зафиксирована для воспроизведения.

### 7.1 Роли

- **Owner (project owner)** — держит цели, принимает архитектурные решения, кликает в AppSheet руками, проверяет результаты. Единая точка валидации.
- **Opus (стратег)** — память проекта, архитектурные решения, обоснования, подготовка брифингов, сборка бэкапов.
- **Sonnet (исполнитель)** — пошаговая реализация в UI AppSheet с короткой памятью. Получает брифинг — выдаёт отчёт.

### 7.2 Артефакты-носители памяти

- **Бэкап sub-general** (этот файл) — состояние ветки, передаётся в новую сессию первым сообщением
- **Брифинги** для Sonnet — узкие инструкции на один рабочий день
- **Отчёты Sonnet** — фактические результаты + новые best practices

### 7.3 Стандарт брифинга

Структура (обязательная):
1. Шапка (дата, проект, ссылки)
2. Формат работы — 2–3 микрошага за раз, без размотки, без вступлений/послесловий
3. Терминология (primary/menu и т.п.)
4. Best practices проекта (накопленный список)
5. Контекст задачи в одну минуту
6. Где остановились (точное состояние)
7. План на день — задачи по шагам
8. Что НЕ входит в задачу
9. Формат отчёта (всегда `.md`)

### 7.4 Гранулярность шагов в брифинге
2–3 микро-действия в одном указании. Не «весь логический шаг сразу» (Sonnet размотает) и не «по одному клику» (медленно). Связка из 2–3 UI-действий → пауза на скриншот.

### 7.5 Стандарт отчёта Sonnet

1. Что сделано по задачам
2. Финальные настройки (поля, Slices, views, формулы)
3. Отклонения от брифинга и причины
4. Незакрытые хвосты (переезжают в брифинг следующего дня)
5. Скриншоты ключевых результатов
6. Дополнения в best practices, если выявились новые

### 7.6 Вечернее резюме
Owner может попросить Opus в конце дня собрать короткое резюме (что сделано / не сделано / решения / следующий день). Полтора абзаца, не простыня. Цель — разгрузить рабочую память owner перед сном.

### 7.7 Закрытие ветки
Признаки: Opus начинает забывать ранее принятые решения, дает «простыни» вместо коротких ответов, путает терминологию. Тогда — собирать новый бэкап и открывать чистую ветку.

## 9. Стратегия восстановления из бэкапа

**Контекст:** AppSheet и Sheets — две независимые системы. AppSheet привязан к Sheets-файлу по ID. При восстановлении из xlsx-бэкапа создаётся новый Sheets-файл с новым ID — это требует перепривязки таблиц в AppSheet.

### Сценарии

**A. Откат через историю версий Google Sheets** (ID не меняется)
- AppSheet продолжает работать сам, ничего перенастраивать не нужно
- Минимальное вмешательство

**B. Восстановление из xlsx-бэкапа в новый Sheets-файл** (новый ID)
- В AppSheet каждую таблицу — Data → таблица → Source path → перепривязать на новый файл
- Оценка времени: ~10 минут (требует репетиции — следующий шаг плана)

**C. Восстановление через `File → Import → Replace`** в существующий Sheets
- ID сохраняется, AppSheet работает
- Самый чистый путь, если оригинальный файл доступен

**D. Удаление AppSheet-приложения**
- AppSheet хранит App History, можно откатиться
- Если безвозвратно — пересборка по бэкапам v4 + отчётам Sonnet (часы работы, но возможно)

### Регулярное резервное копирование (рекомендации)
- Экспорт Sheets в xlsx раз в неделю/месяц (наша версия v7)
- `File → Make a copy` в Drive — независимая копия с другим ID
- AppSheet App History — снимки перед серьёзными изменениями
- После старта системы — Apps Script для автобэкапа в архивную папку

### Тест восстановления (следующий шаг плана)
**Цель:** отрепетировать процедуру B на копии до того, как Apps Script начнёт писать в `charges`.

**Процедура теста:**
1. Copy App в AppSheet → клон боевого приложения

## 12. Открытые вопросы и backlog

### Открытые архитектурные вопросы (решение отложено осознанно)

**12.1. Обнуление баланса по учебным годам.**
Сейчас модель — сквозное накопление за всё время. Альтернатива — закрытие учебного года с переносом сальдо в `opening_balances` следующего года. Решение принимается **после первого полного цикла эксплуатации**, на основе реального опыта (производительность AppSheet, психология восприятия долгов, обработка ушедших учеников).

**12.2. Семантика «свободные средства».**
В дашборде формула `[поступления_год] - [расходы_год]` — это годовой остаток. По брифингу подразумевался месячный. Возможно, нужны оба показателя. Решение — после старта системы, на реальных данных.

**12.3. Тип `charges[period]` в AppSheet.**
В разделе 10 v3 зафиксировано «period как Number YYYYMM». Фактически AppSheet видит его как Enum/Text (MAX не работает, требуется NUMBER() для сравнений). Возможен рефакторинг на чистый Number — отдельная задача после старта.

**12.4. Динамический список периодов в Enum-фильтрах.**
Сейчас захардкожен до 202701 (запас на 9 месяцев). Через ~6 месяцев решить: либо плановое продление, либо переход на динамический список через valid_if = SELECT(charges[period], TRUE).

### Backlog задач (после старта)

- Аналитика расходов по категориям — отдельный лист `analytics` в Sheets с QUERY/PIVOT, владельческий инструмент. Параметры (категория, глубина, гранулярность месяц/неделя) — в ячейках. К AppSheet не подключается.
- Скилл «Подготовка брифинга для исполнителя» — формализация шаблона из раздела 7. Делается, когда накопится 5+ брифингов.
- Group by категории в Deck-списках — Deck не поддерживает, требует отдельного UX-решения.
