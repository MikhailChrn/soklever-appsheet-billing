## §1. Контекст

Все три типа платежа в системе сейчас работают одинаково — увеличивают «сосуд» поступлений. Но **взаимозачёт** — это особый случай: семья гасит долг не деньгами, а услугой/материалом. С точки зрения денежного потока — это **одновременно поступление и расход на ту же сумму**.

Сейчас оператор должен помнить: «отметил взаимозачёт — не забудь внести парный expense». На практике забывается. В конце месяца «деньги в кассе» расходятся со «всего получено».

**Цель:** в момент сохранения `payment` с `payment_type = "Взаимозачёт"` автоматически открывать форму создания `expense` с предзаполненными полями. Оператор подтверждает или правит — и сохраняет.

**Что НЕ закладываем (§1.3 «суровая правда без шума»):**
- Защита от Cancel формы expense. Если оператор закрыл форму без сохранения — парного расхода не будет. Принимается как ограничение, не покрывается.
- Связка `payment ↔ expense` через FK. Сохраняем как два независимых документа. Если потребуется отчёт «взаимозачёты по семьям» — реализуем позже.
- История изменения цепочки (если payment удалили — expense остаётся).

**Паттерн:** SKILL Q.11 (LINKTOFORM vs LINKTOROW), Q.13 (Form Saved event как триггер транзакции).

## §2. Предварительная сверка состава payments

### §2.1. Проверка наличия `payment_type`

В `ARCHITECTURE.md §2.6` состав `payments` указан как:
```
payment_id | student_id | amount | payment_date | comment | class_group
```

`payment_type` в этом списке **не упоминается**, но в `GLOSSARY.md §2.8` есть «Типы платежа: Наличные, Перевод, Взаимозачёт». Скорее всего колонка существует в Sheets, но пропущена при составлении §2.6.

**§2.1.1. Sonnet должен:**
1. Открыть Google Sheets, лист `payments`. Зафиксировать **точный** состав колонок (порядок, имена).
2. В AppSheet → Data → payments — зафиксировать список колонок, их типы и какие из них VC.
3. Сравнить с §2.6 ARCHITECTURE. Расхождения вынести **в отчёт** для актуализации.
4. Если `payment_type` **существует**: убедиться, что Type = Enum, значения `Наличные`, `Перевод`, `Взаимозачёт`. Если значения отличаются — НЕ ПРАВИТЬ, спросить владельца через стратега.
5. Если `payment_type` **отсутствует**: остановиться, доложить стратегу. Создание новой колонки — отдельное решение владельца.

## §3. Архитектурное решение

### §3.1. Триггер

Сохранение `payment` с `payment_type = "Взаимозачёт"`. Событие — **Form Saved** на форме `payments_Form` (или slice-Form, если так настроено).

### §3.2. Действие

При срабатывании события — открыть форму `expenses_Form` через `LINKTOFORM` с предзаполненными полями:
- `amount` = `[amount]` (та же сумма, что в payment).
- `period` = период из `payment_date` (см. §3.4 ниже).
- `category_id` = категория «Прочее» (см. §3.5).
- `expense_date` = `[payment_date]`.
- `description` = `CONCATENATE("Взаимозачёт: ", [student_id].[_form_header_students])` или похожая строка (см. §3.6).

### §3.3. Что НЕ заполняется автоматически

- Поля, которые оператор обязан подтвердить руками. По текущему составу expenses — больше нет обязательных полей кроме указанных. Если в `expenses` есть `created_by` или другие — Sonnet решит при сверке состава.

### §3.4. Формула period из payment_date

Период expense — это `YYYYMM` от `payment_date`:
```
CONCATENATE(YEAR([payment_date]), IF(MONTH([payment_date]) < 10, "0", ""), MONTH([payment_date]))
```

**Важно:** `expenses[period]` — Ref → ref_periods. LINKTOFORM передаёт значение как строку, AppSheet привязывает к ref-таблице по совпадению.

### §3.5. Имя категории «Прочее»

Sonnet должен:
1. Открыть `expense_categories` в Sheets.
2. Найти строку с категорией «Прочее» (или максимально близким именем).
3. Зафиксировать её `category_id`.
4. Использовать этот `category_id` в LINKTOFORM (не строку «Прочее», а именно ID).

**Если категории «Прочее» нет** — остановиться, доложить стратегу.

### §3.6. Описание (description)

Текст в `description` помогает потом разобрать список расходов. Предлагаемый формат:
```
"Взаимозачёт: " + ФИО ученика
```

Если в `expenses` нет колонки `description` — пропустить (категория и сумма всё опишут).

## §4. Реализация — пошагово

### §4.1. Где разместить логику запуска LINKTOFORM

Два варианта, **владельцу решать через стратега до старта**:

**Вариант А.** Bot с триггером **Form Saved on payments**:
- Event: Form Saved on payments_Form.
- Condition: `[payment_type] = "Взаимозачёт"`.
- Process step: `Run a task → Send a notification → no` (это не подходит) **или** `Run a task → Webhook` (нет, тоже не то).

LINKTOFORM из bot **не запустить** напрямую — bot выполняется на сервере и не может открыть UI у пользователя. Этот вариант **не работает**, отбрасываем.

**Вариант Б (рабочий).** Action на `payments`:
- Action `payments__open_barter_expense`.
- `For a record of this table:` `payments`.
- `Do this:` `App: go to another view within this app`.
- `Target:` `LINKTOFORM("expenses_Form", ...все поля...)`.
- `Only if this condition is true:` `[payment_type] = "Взаимозачёт"`.
- `Position:` `Hide` (не нужна как ручная кнопка).
- **Привязать к Form Saved event** формы `payments_Form`:
  - View `payments_Form` → Behavior → Event Actions → Form Saved → выбрать `payments__open_barter_expense`.

При сохранении платежа: если type = Взаимозачёт → action срабатывает → открывается форма expense. Если type иной — Show_If возвращает false → action не делает ничего → пользователь возвращается к списку платежей (стандартное поведение Form Saved).

**Делаем Вариант Б.** Если в текущей версии AppSheet поле Form Saved Action не работает как ожидается — Sonnet докладывает стратегу.

### §4.2. Action `payments__open_barter_expense`

Действие создать с такими параметрами:

| Поле | Значение |
|---|---|
| Action name | `payments__open_barter_expense` |
| For a record of this table | `payments` |
| Do this | `App: go to another view within this app` |
| Target | См. §4.3 |
| Only if this condition is true | `[payment_type] = "Взаимозачёт"` |
| Position | `Hide` |
| Confirmation | OFF |

### §4.3. Target формулы

```
LINKTOFORM(
  "expenses_Form",
  "amount", [amount],
  "expense_date", [payment_date],
  "period", CONCATENATE(YEAR([payment_date]), IF(MONTH([payment_date])<10, "0", ""), MONTH([payment_date])),
  "category_id", "<ID-категории-Прочее-из-§3.5>",
  "description", CONCATENATE("Взаимозачёт: ", [student_id].[_form_header_students])
)
```

**Точное имя view формы expenses** — Sonnet проверяет. Может быть `expenses_Form`, `Add_expenses_Form` или что-то ещё в зависимости от системного именования. Зафиксировать.

**Точные имена колонок** в `expenses` — Sonnet проверяет по составу (§4.5). Если в expenses нет `description` — убрать строку из LINKTOFORM.

### §4.4. Привязка к Form Saved

UX → Views → `payments_Form` → Behavior → **Form Saved** → выбрать `payments__open_barter_expense`.

### §4.5. Сверка состава expenses

Sonnet должен зафиксировать в отчёте: точный состав колонок `expenses` на момент работы. По текущим знаниям (ARCHITECTURE §2.7):
```
expense_id | amount | period | expense_date | category_id | description | created_by
```

Если состав отличается — отметить в отчёте, LINKTOFORM подстроить.

## §6. Чек-лист тестирования

1. **Тест-1:** создать `payment` с `payment_type = "Наличные"`. Сохранить. Ожидаем: возврат к списку платежей, форма expense НЕ открывается.
2. **Тест-2:** создать `payment` с `payment_type = "Перевод"`. Сохранить. Ожидаем: то же, форма expense НЕ открывается.
3. **Тест-3:** создать `payment` с `payment_type = "Взаимозачёт"`, сумма 5000, дата сегодня, ученик любой. Сохранить. Ожидаем: открывается форма expense с предзаполненными полями (amount=5000, expense_date=сегодня, period=202605, category=Прочее, description содержит ФИО ученика). Оператор может ничего не менять — нажать Save.
4. **Тест-4:** после Тест-3 проверить:
   - В `payments` появилась новая запись (Взаимозачёт).
   - В `expenses` появилась новая запись (Прочее, 5000, сегодня).
   - Дашборд: «всего получено» включает новый payment; «всего израсходовано» включает новый expense.
5. **Тест-5:** повторить Тест-3, но в форме expense нажать **Cancel** (не Save). Ожидаем: payment сохранён, expense не создан. Зафиксировать в отчёте (это ограничение, не баг).
