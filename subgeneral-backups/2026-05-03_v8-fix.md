## §F2. Дашборд — все 9 формул дословно

Все VC живут в таблице `settings`. Тип у всех — Number (кроме хелпера `dash_пред_период`, тоже Number).

### §F2.1. Блок 1 — Живая зона (открытый период)

**`settings[dash_открытый_поступления]`:**
```
SUM(SELECT(payments[amount],
  AND(
    YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
    MONTH([payment_date]) = NUMBER(RIGHT(ANY(settings[current_period]), 2))
  )
))
```

**`settings[dash_открытый_расходы]`:**
```
SUM(SELECT(expenses[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**`settings[dash_открытый_начислено]`:**
```
SUM(SELECT(charges[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**Архитектурное наблюдение:** три формулы используют **разные** способы определить «текущий период»:
- Поступления — по `payments[payment_date]` через `YEAR()`+`MONTH()`. Потому что у `payments` нет `period` (Р-1).
- Расходы и начисления — по `[period] = current_period`. Потому что у `expenses` и `charges` `period` есть.

Это **намеренно**, не баг. Следствие: платёж, проведённый 30.04 с реальной датой 30.04, попадает в апрельские поступления. Платёж, проведённый бэк-датой за март — в мартовские. Так и должно быть по модели сосудов.

### §F2.2. Блок 2 — Долги

**`settings[dash_долги_количество]`:**
```
COUNT(SELECT(families[family_id], [family_overdue_debt] > 0))
```

**`settings[dash_долги_сумма]`:**
```
SUM(SELECT(families[family_overdue_debt], [family_overdue_debt] > 0))
```

### §F2.3. Блок 3 — Твёрдая земля

**`settings[dash_земля_получено]`:**
```
SUM(SELECT(payments[amount],
  AND(
    [payment_date] >= ANY(settings[academic_year_start]),
    OR(
      YEAR([payment_date]) < NUMBER(LEFT(ANY(settings[current_period]), 4)),
      AND(
        YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
        MONTH([payment_date]) < NUMBER(RIGHT(ANY(settings[current_period]), 2))
      )
    )
  )
))
```

**`settings[dash_земля_израсходовано]`:**
```
SUM(SELECT(expenses[amount],
  AND(
    NUMBER([period]) >= YEAR(ANY(settings[academic_year_start])) * 100 + MONTH(ANY(settings[academic_year_start])),
    NUMBER([period]) < NUMBER(ANY(settings[current_period]))
  )
))
```

**`settings[dash_земля_свободные]`:**
```
[dash_земля_получено] - [dash_земля_израсходовано]
```

## §F3. Процедура закрытия периода (§О-1, развёрнуто)

Восстановлено владельцем по памяти 03.05.2026, что было принято в #7 (на момент v7).

**Кнопка «закрыть период» делает ровно одно действие — генерирует регулярные начисления на следующий месяц.**

**Конкретно:**

1. Читает `settings[current_period]` (например, `202605`).
2. Для каждой строки из `student_tariffs`, у которой:
   - связанный `tariffs[tariff_type] = "Регулярный"`,
   - связанный `students[status] = "Активен"` *(условие добавлено в #9 после обнаружения поля `status`)*,
   создаёт одну строку в `charges`:
   - `student_id` = `student_tariffs[student_id]`
   - `tariff_id` = `student_tariffs[tariff_id]`
   - `charge_type` = `"Регулярный"`
   - `period` = следующий период из `ref_periods` (например, `202606`)
   - `amount` = `[tariff_id].[amount]` на момент создания (снимок, см. Р-8)
   - `charge_date` = `TODAY()`
3. **Сдвиг `current_period` явно не нужен** — он вычисляется формулой `INDEX(SORT(SELECT(charges[period], [tariff_id].[tariff_type] = "Регулярный"), TRUE), 1)`, то есть обновится сам после появления новых charges с `period = 202606`.

**Что кнопка НЕ делает:**
- Не создаёт снимок должников (Р-5: долги динамические).
- Не сохраняет состояние «закрытого» периода.
- Не помечает старые `charges` как закрытые.
- Не трогает разовые начисления.

**Обоснование** (Р-1, Р-5, Р-7, §2.1, §2.2): модель сосудов + динамический долг = снимочная логика не нужна. После сдвига `current_period` старый месяц автоматически становится твёрдой землёй.

**UX (решено в #9):**
- Кнопка живёт в общем view `tariffs` (справочник тарифов).
- Одна кнопка на все тарифы и всех учеников сразу.
- Confirmation message: «Вы точно хотите закрыть период YYYYMM? Следующий открытый период будет YYYY(MM+1).» — где YYYY(MM+1) берётся из `ref_periods` как следующий по списку.
- После выполнения — справка: «Создано начислений N. Выявлено должников T.» — где T = `dash_долги_количество` после сдвига (общее число должников, не прирост).

**Платформа:** AppSheet Bot, триггерится Action из view `tariffs`. Не Apps Script.

**Регламент запуска:** вручную, 28–31 числа.

## §F4. Формулы долга — дословно

### §F4.1. `students[debt]` — текущая задолженность ученика

Тип: Number, VC.

```
SUM(SELECT(charges[amount], [student_id] = [_THISROW].[student_id]))
- SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
- SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id]))
```

**Семантика:** баланс сосуда ученика. Может быть положительным (должен), нулевым (квит), отрицательным (переплатил). Это и есть «уровень в сосуде» из §2.1.

### §F4.2. `students[overdue_debt]` — дебиторская задолженность ученика

Тип: Number, VC.

```
IFS(
  SUM(SELECT(charges[amount],
    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))
    )
  ))
  - SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id]))
  > 0,
  SUM(SELECT(charges[amount],
    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))
    )
  ))
  - SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id])),
  TRUE,
  0
)
```

**Семантика:** только просроченная часть. Всегда ≥ 0. Это то, что делает должником. Эквивалентно `MAX(0, ...)`, написанному через IFS с продублированным выражением.

**Структурное замечание:** выражение продублировано (внутри условия и внутри ветки-результата). Если потребуется править формулу — править в **обоих** местах. Кандидат на переписывание через `MAX(0, ...)` в будущем — но не сейчас, формула работает корректно.

### §F4.3. `families[family_debt]` — текущая задолженность семьи

```
SUM(SELECT(students[debt], [family_id] = [_THISROW].[family_id]))
```

**Без обрезки нулём.** Переплата одного ученика **компенсирует** долг другого внутри семьи.

### §F4.4. `families[family_overdue_debt]` — дебиторская задолженность семьи

```
SUM(SELECT(students[overdue_debt], [family_id] = [_THISROW].[family_id]))
```

**Каждый `overdue_debt` уже обрезан нулём.** Переплата одного ученика **не** компенсирует просрочку другого внутри семьи.

**Это намеренная асимметрия** для двух показателей. Подтверждено владельцем 03.05.2026.

### §F4.5. Что значит «строго раньше current_period»

Это фильтр на `charges[period]`, не на `payments`. Платежи в формуле долга учитываются **все**, без фильтра по дате. Это правильно по модели сосудов: деньги падают в общий сосуд ученика, без привязки к месяцу.

Сравнение `[period] < current_period` сделано через `NUMBER([period]) < NUMBER(ANY(settings[current_period]))` — обе колонки после #6 это Ref на `ref_periods` со значениями Text вида YYYYMM. Через `NUMBER()` приводится к числу для сравнения.

### §F4.6. `opening_balances` в формуле долга

Обе формулы (`debt` и `overdue_debt`) **вычитают** `SUM(opening_balances[amount])` для ученика. Эта таблица — фундамент модели сосудов на момент запуска биллинга. Без неё ученик с долгом до запуска системы выглядел бы квитом.

См. §F5.6 — реальный состав таблицы.

## §F7. Уточнения архитектурных решений

### §F7.1. Уточнение к Р-6

Формула долга:
- Учитывает **все** платежи ученика (без фильтра по дате) и **все** `opening_balances` ученика.
- Фильтр по периоду — **только** на начислениях (`charges`).
- В `family_debt` переплаты учеников **компенсируют** долги внутри семьи.
- В `family_overdue_debt` переплаты **не** компенсируют просрочки (каждый `overdue_debt` уже обрезан нулём).

### §F7.2. Уточнение к §О-1 (закрытие периода)

При генерации `charges` фильтровать `student_tariffs` по двум условиям:
- `[tariff_id].[tariff_type] = "Регулярный"`,
- `[student_id].[status] = "Активен"`.

Поле `status` обнаружено в `students` в #9. Без фильтра по нему скрипт сгенерирует начисления выбывшим ученикам.

### §F7.3. Уточнение к §Р-9 (дашборд)

В `settings` существует VC `dash_пред_период` (см. §F2.4). В §Р-9 v8 не упомянут.

## §F8. Открытый вопрос: процесс наследования между сессиями

**Проблема, увиденная в #9:** бэкап как формат накапливает потери. За 6 поколений (v3 → v8) важные архитектурные детали (формулы долга, состав таблиц, процедура закрытия периода) либо не попали в текст, либо «усохли» до ссылок.

**Корни проблемы:**
1. Каждый предок собирает бэкап «на своё усмотрение» — что считать важным для следующего.
2. Пересказы своими словами теряют точность. Формула, переписанная как «суммирует начисления за прошлые периоды минус все платежи» — не содержит достаточной информации, чтобы восстановить её обратно.
3. Владелец — единственный физический хранитель проекта между сессиями — не имеет ресурса вычитывать каждый бэкап с проектом в руках.

**Гипотеза о решении** (мысль ещё SubGeneral #8, развита в #9):

`formulas-guide` (запланирован в #10 как «синхронизация эталонов») должен превратиться из косметического шага в **обязательный артефакт каждого бэкапа**. Не словесный пересказ архитектуры, а:
- Дословные App formulas всех VC.
- Реальные составы всех таблиц (как §F5).
- Cross-reference: какая формула на какие колонки опирается.

Тогда даже если предок упустит нюанс — формула сама себя задокументирует.

**Что делать:** обсудить с владельцем после закрытия #7 и #8. Возможно, перенести #10 ближе к началу, объединив его с регламентом бэкапа. Не сейчас, дедлайн 1 июня жмёт.

**Не забыть.** Иначе следующий SubGeneral начнёт с того же расследования, что и #9.
