# Архитектурная сессия по ТД-1 — Решения

**Дата:** 02.05.2026
**Ветка:** SubGeneral #8
**Версия приложения на момент сессии:** 1.000337
**Участники:** SubGeneral #8, Владелец
**Входные данные:** отчёты br05 + br05.1, добивка по `payments[period]`, правка П.3 (`11класс` → `11 класс`).

---

## Постановка проблемы

Значения периодов (YYYYMM) и классов разбросаны по проекту в виде свободных Enum'ов: 6 мест по периоду, 3 по классу, плюс упоминания в формулах, slice'ах, Views. При добавлении нового значения легко забыть один из углов — возникает рассинхронизация. Уже зафиксирован случай: `settings[filter_charges_class]` имел `"11класс"` (без пробела) и `AllowOtherValues: true`, тогда как два других классовых Enum'а — `"11 класс"` и `false`. Фильтр расходов по 11-му классу был сломан в проде.

Главное требование владельца: **система должна работать без него**. Если проектом займётся другой человек, структура должна объяснять себя сама.

---

## Решения

### 1. Справочник `ref_periods`

- Одна колонка: `period` (YYYYMM, тип Number или Text — уточнить при реализации, ключ).
- Заполняется **вручную**, протягиванием в Google Sheets, на 10 лет вперёд (~120 строк, разовое действие).
- Без дополнительных атрибутов (название, статус, тип, флаг закрытия). Добавятся, если когда-нибудь понадобятся.
- **Без автогенерации Bot'ом** — ручное заполнение прозрачнее и не имеет скрытых точек отказа.

**Шесть нынешних Enum'ов переводятся в Ref на `ref_periods`:**

- `charges[period]`
- `expenses[period]`
- `settings[current_period]`
- `settings[filter_period]`
- `settings[filter_payments_period]`
- `settings[filter_charges_period]`

### 2. Ограничение выпадающего списка периодов — Valid_If по месту

Список из 120 значений в выпадающем списке неудобен. Ограничение задаётся **на уровне колонки**, индивидуально под контекст использования:

- В формах ввода (`expenses_Form`, `payments_Form`, `charges_Form`) — окно вокруг текущего периода (точные границы — в брифинге #6).
- В фильтрах архива (`filter_period`, `filter_charges_period`, `filter_payments_period`) — все периоды, в которых есть данные.
- В `settings[current_period]` — выбора нет, значение вычисляется существующей формулой.

Сам справочник `ref_periods` остаётся плоским — фильтрация на стороне потребителя.

### 3. Справочник `ref_classes`

- Одна колонка: `class_group` (Text, ключ).
- Заполняется **вручную**, ~13 строк.
- Без атрибутов.

**Три нынешних Enum'а переводятся в Ref:**

- `students[class_group]`
- `settings[filter_charges_class]`
- `settings[filter_payments_class]`

Рассинхронизация значений (`"11класс"` vs `"11 класс"`) становится физически невозможной.

Valid_If для классов не нужен — 13 значений помещаются в выпадающий список целиком.

### 4. Сопутствующие хвосты — включаются в #6

**4.1.** Переименовать `tariffs[period]` (Enum со значениями `"Месяц"/"Разовый"`) — устранить семантический омоним с YYYYMM-периодом. Конкретное имя — в брифинге.

**4.2.** Удалить ссылки-фантомы на `period` в Views, где колонки нет в схеме:
- `payments_Form` → ColumnOrder.
- `платежи_фильтр_список` → SecondaryDeckHeaderColumn.

**4.3.** Удалить колонку `expenses[period_num]` (Spreadsheet formula, дублирует `expenses[period]` как Number). Формулы, использующие `NUMBER([period])`, оставляем без изменений.

**4.4.** Дубль `_ComputedName` (`students`) и `_student_name` (`student_tariffs`) — объединить в один источник.

## Что НЕ делаем

- Атрибуты периода (название «Апрель 2026», флаги «открыт/закрыт», тип «учебный/летний») — школа в них пока не нуждается.
- Автогенерацию периодов через Bot.
- Историю классов у учеников (`students[class_group]` остаётся свойством «в моменте», переход 1→2 перезаписывает значение).
- Не лезем в Actions, Bots, Format Rules. Если там окажутся ссылки на старые Enum-значения — выловим при тестировании #6.

## Регламентные следствия для брифинга #6

- Перед стартом — **обязательный бэкап** по §Р-15 (Sheets `ancestor-v7` + копия приложения `subgeneral-v8`).
- Порядок реализации в #6:
  1. Создать справочники `ref_periods` и `ref_classes`, заполнить.
  2. Хвост 4.1 (переименование `tariffs[period]`) — делается **первым**, до Ref-перевода, чтобы не цеплять чужой `[period]`.
  3. Хвосты 4.2, 4.3 — в любом порядке до Ref-перевода.
  4. Перевод колонок в Ref (по одной таблице за раз, с проверкой, что приложение поднимается после каждого шага).
  5. Настройка Valid_If по месту использования.
  6. Хвост 4.4 — финальной зачисткой.
- Тестировать после каждого крупного шага: фильтры дашборда, формы ввода, дашборд-формулы.
- Отдельно проверить, что `settings[current_period]` со своей формулой `INDEX(SORT(...))` корректно работает с Ref-колонкой `charges[period]`.

## Регламентные находки этой сессии (для бэкапа v8)

- **Sonnet может в нарушение брифинга записать в Sheets**, даже когда брифинг прямо требует «никаких изменений». В будущих брифингах фиксировать явно: «никаких формул в живые листы Google Sheets даже временно для проверки».
- **AppSheet падает при расхождении числа колонок Sheets vs схемы**, причём «колонкой» Sheets считает любой столбец с непустой формулой/форматированием — даже если шапка визуально пустая. Лечение — очистка Sheets (Delete column на мусор), Regenerate Schema **не требуется**.
- **Ни Regenerate Schema, ни очистка Sheets не двигают версию приложения AppSheet.** Версия меняется только от изменений в редакторе AppSheet.
- **AppSheet App Documentation** не выгружает колонки со `Show? = false`. Аудит по Documentation может пропустить скрытые колонки — это надо учитывать.
