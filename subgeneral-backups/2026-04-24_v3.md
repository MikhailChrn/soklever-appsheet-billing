## 0. Что такое sub-general и чем отличается от general

**general** — описывает проект СО Клевер полностью: Java + React стек, модули Auth, Books, Billing, My, этапы 1–3, контуры INTERNAL/OFFICIAL, архитектура целевого продукта.

**sub-general (этот бэкап)** — Этап 0 («Google-прародитель»): упрощённый учёт на стеке Google Sheets + AppSheet + Apps Script. Цель — сейчас же дать супруге-оператору рабочий инструмент для учёта должников и систематизации расходов, пока основной продукт разрабатывается.

Sub-general самодостаточен. Связь с general минимальна: данные, накопленные в Этапе 0, в будущем могут быть мигрированы в полноценную БД, но это не решается сейчас.

## 4. Принятые решения по следующему шагу — закрытие периода

Обсуждено и зафиксировано 24.04.2026.

### 4.1 Модель активного периода

В системе всегда есть **ровно один активный период** — месяц, за который:
- начисления уже созданы,
- платежи принимаются,
- непогашенные суммы ещё **не** считаются дебиторкой.

**Закрытие периода** — это событие, после которого:
- непогашенные начисления активного месяца превращаются в дебиторку,
- создаются начисления за следующий месяц,
- следующий месяц становится новым активным.

**Платёж имеет свой period.** Можно принять платёж за любой месяц (даже задним числом).

**Формально «активный/закрытый» нигде не хранится** — это производное понятие. Активный период = максимальный period среди регулярных charges. Дебиторкой считается то, что не оплачено за периоды меньше активного.

### 4.2 Что делает кнопка «Закрыть период»

1. Находит **максимальный period среди регулярных charges** (charge_type ≠ «Единоразовый» — берём только от регулярных тарифов).
2. Считает следующий period (+1 месяц, с переходом года).
3. Создаёт charges за новый period для всех активных учеников, у которых есть действующие регулярные тарифы.
4. Ставит charge_type = «Авто».

**Всё.** Никаких флагов, никаких snapshot'ов. Модель данных не меняется.

### 4.3 Действующие тарифы

Смена тарифа в середине месяца не допускается — только со следующего. Поэтому логика проще обычной:

Тариф действует для period P, если привязка в student_tariffs существует и не закрыта на 1-е число месяца P.

Условие: `assigned_from <= [1-е число P]` И (`assigned_to` пусто ИЛИ `assigned_to >= [1-е число P]`).

### 4.4 Идемпотентность

**Не реализуется.** Если оператор случайно нажмёт кнопку дважды — создадутся начисления за ещё один будущий период. Это станет видно («появились начисления за период, платежи по которому никто не принимал») — оператор откатывает вручную.

### 4.5 Расположение кнопки

В **AppSheet** (Action, вызывающая webhook на Apps Script). Оператор принимает платежи там же — логично не переключать контекст.

### 4.6 Триггер — только ручной

Кнопка. Никаких time-driven triggers. Оператор сама решает, когда закрывать — «когда все платежи за месяц проведены».

## 5. Стратегия старта системы (апрель 2026)

### Шаг 1. Подготовка данных
1. Очистить тестовые данные (Алиса, Миша, Даша, Артём) из всех таблиц.
2. Завести реальные семьи и учеников.
3. Завести реальные тарифы.
4. Назначить тарифы ученикам (student_tariffs), `assigned_from` = 2026-04-01.
5. Внести opening_balances на 1 апреля 2026 (долги/переплаты с «тетрадочного» учёта).

### Шаг 2. Первые начисления за апрель
Одноразовый запуск того же скрипта с параметром `period = 202604`. Скрипт создаёт charges за апрель для всех активных учеников.

(Вручную — ~100 начислений, долго и с опечатками. Скрипт — одна функция, параметризуемая.)

### Шаг 3. Работа в течение апреля
Оператор принимает платежи в AppSheet. Все платежи идут с `period = 202604`.

### Шаг 4. Конец апреля — первое закрытие
Оператор нажимает кнопку «Закрыть период». Скрипт:
- видит максимальный регулярный period = 202604;
- создаёт charges за 202605 для активных учеников;
- апрель закрыт (концептуально: теперь непогашенные charges 202604 — дебиторка).

### Шаг 5. Дальше — месячный цикл
Повторяется: месяц работы → кнопка → следующий месяц.

### Устойчивость к опозданиям
Если оператор забыла нажать кнопку в конце апреля и спохватилась в мае — ничего не ломается. Платежи за май уже принимаются (с period 202605), но charges за май ещё нет. Оператор нажимает кнопку → скрипт видит max = 202604 → создаёт 202605. Платежи «находят свои начисления». Работает.

## 6. Параметры скрипта автоначислений

**Имя функции (рабочее):** `closePeriod()` или `generateChargesForNextPeriod()`
**Параметр:** необязательный `targetPeriod` (Number, YYYYMM). Если не задан — вычисляется как «max регулярных + 1». Если задан — используется как есть (для первичной инициализации).

**Псевдокод:**
```
function closePeriod(targetPeriod) {
  if (targetPeriod === undefined) {
    maxPeriod = max(charges.period, where tariff.tariff_type = "Регулярный")
    targetPeriod = nextMonth(maxPeriod)
  }
  
  activeStudents = students where status = "Активен"
  
  for each student in activeStudents:
    assignments = student_tariffs where student_id = student.id
                                    and tariff.tariff_type = "Регулярный"
                                    and assigned_from <= firstDay(targetPeriod)
                                    and (assigned_to is empty OR assigned_to >= firstDay(targetPeriod))
    
    for each assignment in assignments:
      create charge {
        student_id: student.id,
        period: targetPeriod,
        tariff_id: assignment.tariff_id,
        amount: tariff.amount,
        charge_type: "Авто",
        description: "Автоначисление за " + periodDisplay(targetPeriod),
        created_at: now()
      }
}
```

**Что нужно решить при реализации:**
- Webhook из AppSheet → Apps Script: через Doget/Dopost endpoint или через прямой Apps Script trigger.
- Формат charge_id — автогенерация или UNIQUEID (в AppSheet уже используется UNIQUEID).
- Обработка ошибок — отправлять ли уведомление о результате в AppSheet.
- Тестирование: сначала запустить на копии Sheets с тестовыми данными.

## 10. Ключевые технические решения — сводка

### Google Sheets
| Задача | Решение |
|---|---|
| Раскрытие формул по колонке | ARRAYFORMULA |
| SUMIFS по парам значений построчно | MAP + LAMBDA |
| Накопительные суммы с диапазоном | SUMPRODUCT (не SUMIFS со строкой `">="&val`) |
| Выборка с условиями | FILTER (с одинаковой длиной диапазонов) |
| Группировка и агрегация | QUERY (для числовых period — служебные колонки period_num) |
| Двойной JOIN | VLOOKUP(VLOOKUP(...)) |
| Устойчивость к типу period при импорте | `*1` и `IFERROR(...*1; 0)` |

### AppSheet
| Задача | Решение |
|---|---|
| ID генерация | UNIQUEID() |
| Долг ученика | virtual column (SUM charges - SUM payments - SUM opening_balances) |
| Долг семьи | virtual column (SUM по детям) |
| Ограничение доступа OPERATOR | Security Filter по колонке created_by |
| Автозаполнение created_by | USEREMAIL() |
| Не подключать debts/dashboard | Формулы ARRAYFORMULA/QUERY не читаются AppSheet |
| Подсветка долгов | Format Rule → красный при debt > 0 |

### Структурные
| Решение | Обоснование |
|---|---|
| period как Number YYYYMM | Удобнее для QUERY и SUMPRODUCT |
| debts — журнал, не сводка | В AppSheet агрегация делается virtual columns, журнал нужен для контроля |
| Две параллельные системы представления (Sheets + AppSheet) | Разные пользователи, разные задачи |
| Смена тарифа только с начала месяца | Упрощает логику скрипта автоначислений |
