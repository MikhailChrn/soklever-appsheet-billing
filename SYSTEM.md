## §1. Назначение системы

### §1.1. Что это

Система биллинга для частной школы **СО Клевер**. Учёт начислений за обучение, платежей от семей, долгов и расходов школы.

**Этап развития:** Этап 0 — «Google-прародитель». Временное решение на стеке Google Sheets + AppSheet. В перспективе — переход на основной продукт (Java + React, отдельный большой проект).

### §1.2. Кто пользователи

- **Владелец и основной разработчик системы:** Михаил.
- **Главный пользователь:** супруга Михаила — основатель и операционный директор школы. Работает с дашбордом, принимает платежи, контролирует должников, отвечает за финансовую картину.
- **Администраторы школы (роль OPERATOR):** вносят текущие расходы (свои закупки, оплаты педагогам и т.п.). Доступ ограничен одним view.

### §1.3. Дедлайн запуска

**01.06.2026** — система должна работать с реальными данными. Сначала летний цикл (Р-11), потом обычная эксплуатация.

### §1.4. Что система решает

1. Учёт привязок ученик-тариф.
2. Автоматическое начисление платы в начале каждого месяца через закрытие периода.
3. Учёт платежей от семей (модель сосудов, без привязки к конкретному начислению).
4. Подсчёт долгов на лету (текущая задолженность + дебиторская задолженность по Р-6).
5. Учёт расходов школы по категориям и периодам.
6. Подведение финансового итога по закрытым периодам и за учебный год.
7. Дашборд с ключевыми цифрами для руководителя (Р-9).

### §1.5. Что система НЕ решает

- Не учитывает посещаемость, успеваемость, расписание.
- Не общается напрямую с родителями (нет личного кабинета).
- Не делает прогнозы и аналитику «потенциальных» сценариев.
- Не хранит историю изменений тарифов, состава классов и т.п.

## §2. Принципы системы

### §2.1. Модель сосудов

Платежи и начисления работают как два сообщающихся сосуда. Долг = разность уровней. Платёж не привязан к начислению.

→ детали см. `GLOSSARY.md` §1.1.

### §2.2. Твёрдая земля и живая зона

Закрытые периоды — твёрдая земля. Текущий открытый период — живая зона. Должником считается тот, у кого есть долг по твёрдой земле.

→ детали см. `GLOSSARY.md` §1.2.

### §2.3. Суровая правда жизни без шума

Простота важнее «гибкости на потом». Не закладываем механики, которые не приносят пользы прямо сейчас.

→ детали см. `GLOSSARY.md` §1.3.

### §2.4. Параллельные системы представления

- **Sheets** — источник истины для владельца (Михаил).
- **AppSheet** — операторский UI для основного пользователя (супруга) и администраторов (OPERATOR).

Разные инструменты для разных людей. Лист `dashboard` в Sheets к AppSheet **не подключаем**.

## §4. Сущности (бизнес-модель)

Технические детали (колонки, типы, формулы) — в `ARCHITECTURE.md`. Здесь — только смысл.

### §4.1. Семья (family)

Логическая единица оплаты. Имеет название, телефон, email. Долги детей семьи **агрегируются** в семейный долг — он и считается.

### §4.2. Ученик (student)

Конкретный ребёнок в школе. Принадлежит семье, имеет класс, дату начала обучения, статус.

**Статус:**
- `"Активен"` — учится сейчас, при закрытии периода создаются начисления.
- `"Выбыл"` — закончил обучение, новых начислений нет, накопленный долг — за семьёй.

**Когда выбывший полностью рассчитался** — скрывается из основного списка учеников. Карточка остаётся в БД для истории.

### §4.3. Тариф (tariff)

Регулярная стоимость обучения в месяц. На момент 07.05.2026 — 9 базовых тарифов:
- Базовые по классам: «2 класс, база», «3 класс, база», «5 класс, база», «7 класс, база».
- Продлёнки: «Мини», «Стандарт», «Оптимал», «Полный».
- Летний взнос (T009 — `[?? уточнить судьбу — оставить или удалить, см. v10 §7.4]`).
- T010 «Летний тариф» — будет создан в брифинге #08.

### §4.4. Привязка ученик-тариф (student_tariff)

«Ученик X учится по тарифу Y». Один ученик может иметь несколько привязок (база + продлёнка). **Без дат, без истории** (Р-14). Удаление строки = со следующего закрытия начисления по этому тарифу не создаются.

### §4.5. Начисление (charge)

«Ученик должен Z рублей за месяц YYYYMM». Создаётся при закрытии периода (`charge_type = "Авто"`) или вручную (`charge_type = "Ручное"`). Сумма фиксируется на момент создания.

### §4.6. Платёж (payment)

Внесение оплаты от семьи. Привязан к ученику, без поля `period` (Р-1). Работает модель сосудов.

### §4.7. Расход (expense)

Финансовая операция школы. Не связан с учениками. Имеет **период отнесения** (`period`, для дашборда) и **дату факта** (`expense_date`). Категории: Аренда, Педагоги, Коммуналка, Уборка, Хозтовары, Канцтовары, Мероприятия, Оборудование, Прочее.

### §4.8. Открывающие балансы (opening_balances)

Историческая таблица для миграции с другой системы. Позволяет указать «ученик X на момент входа в систему уже должен/переплатил Y рублей».

## §5. Бизнес-процессы

### §5.1. Закрытие периода

**Когда:** в последних числах закрываемого месяца (Р-7). Например, апрель закрываем 28-30 апреля, не 5-10 мая.

**Кто:** ADMIN.

**Шаги:**
1. ADMIN открывает view «скрипты», нажимает «Закрыть период».
2. Confirmation: «закрыть период X, открытый — Y».
3. Бот проверяет: можно ли закрывать (Branch). Если уже закрыто — сообщает «уже закрыто».
4. Создаёт `charges` за следующий период по всем активным привязкам ученик-тариф.
5. Записывает в `settings` дату и сообщение.
6. Дашборд обновляется автоматически.

### §5.2. Регистрация платежа

Через карточку ученика → «+ платёж» → форма (дата, сумма, тип, комментарий).

`payment_date` редактируется (можно проставить задним числом для стыков месяцев), default = TODAY() (Р-4).

### §5.3. Регистрация расхода

Через view «расходы» → `+`. Имеет период отнесения (для дашборда), дату факта, категорию, сумму, получателя, комментарий.

OPERATOR может вносить расходы — видит только свои.

### §5.4. Выбытие ученика

Через карточку ученика → кнопка «Выбытие» → выбор даты → confirmation → транзакция:
- `end_date` = выбранная дата,
- `status` = `"Выбыл"`,
- `class_group` = пусто,
- удаление всех `student_tariffs` ученика.

При следующем закрытии периода — этому ученику начислений уже не будет. Текущее начисление за уже открытый период **не пересчитывается** (начисления не дробим).

→ реализация — pre-#08.1 (планируется 07-08.05.2026).

### §5.5. Просмотр должников

Через view «Должники» (Primary меню). Семьи с `family_overdue_debt > 0`. По каждой — телефон/email для связи, кнопки звонка/email/SMS.

### §5.6. Летний цикл (Р-11)

| Дата | Действие |
|---|---|
| 31 мая | Закрытие мая |
| 1 июня | **Массовая смена тарифа на летний** (через скрипт, брифинг #08) |
| Июнь–август | Обычные циклы закрытия |
| Конец августа | Закрытие августа |
| 1 сентября | **Ручная** пересборка: классы, состав учеников, индивидуальные тарифы |
| 30 сентября | Закрытие сентября, новый учебный год |

### §5.7. Закрытие учебного года (Р-10)

Никакой специальной механики. В августе:
1. Снимаем физический остаток денег.
2. Проводим как `expenses`.
3. Накопительные итоги обнуляются естественным образом.
4. 1 сентября — нулевая твёрдая земля.

## §6. Дашборд (Р-9)

Дашборд для **руководителя школы** (супруги Михаила). Отвечает на 4 вопроса:

1. Сколько денег в казне сейчас?
2. Сколько ещё придёт в этом месяце?
3. Сколько нам должны?
4. Что с расходами в этом месяце?

**Структура — три блока сверху вниз:**

### §6.1. Блок 1. Живая зона (текущий открытый период)

- Поступления за открытый период.
- Расходы за открытый период.
- Всего начислено в открытом периоде.

### §6.2. Блок 2. Долги (твёрдая земля)

- Количество должников-семей.
- Общий долг.

По Р-6: долг считается **только** по периодам строго раньше `current_period`.

### §6.3. Блок 3. Накоплено за учебный год (твёрдая земля)

- Получено в закрытых периодах (с 1 сентября).
- Израсходовано в закрытых периодах.
- Свободные средства = разность.

**Прогнозирование не закладываем** (Р-9, §2.3).

### §6.4. Дополнительная плитка «Последнее закрытие»

Введена в брифинге #07.3. Показывает:
- `last_close_at` — timestamp последнего закрытия.
- `last_close_message` — текст результата (например, «Период 202605 закрыт. Открытый период: 202606. Создано начислений: 6.»).

## §7. Стек и платформа

### §7.1. Хранилище данных

**Google Sheets:** `2026-04-17 soklever-google-ancestor-v6 (main)`. 11 листов:
- 8 рабочих: families, students, tariffs, student_tariffs, charges, payments, expenses, opening_balances.
- 3 служебные: settings, expense_categories, users.

`debts` и `dashboard` листы — рудимент эпохи до AppSheet, к удалению.

### §7.2. Интерфейс и логика

**AppSheet:** приложение `soklever-root-billing v.1.0`. Основной UI для повседневной работы.

**Закрытие периода** — через AppSheet Bot (внутри AppSheet, без Apps Script). Триггер — служебная колонка `settings[last_close_trigger]`.

### §7.3. Окружение

Только облачное (Google + AppSheet). Локальных компонентов нет.
