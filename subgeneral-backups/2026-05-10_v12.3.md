## §1. Что произошло за 10.05.2026

### §1.1. Утро — сбор хвостов и составление pre-#08.3

Владелец прислал перечень из 5 пунктов после ревью UI. Анализ дал три однотипных проблемы (заголовки форм при редактировании показывают «Новый …» вместо «Редактирование …») и две сопутствующие (нет Detail+Edit на карточках charges/payments; рудиментарный тумблер `tariff_type`).

Архитектурное решение от владельца, зафиксировано в `SYSTEM.md` v0.4.3 §4.3 и `GLOSSARY.md` v0.3.2 §2.9: **все тарифы по определению регулярные**, разовые сборы — через `charges[charge_type] = "Ручное"`. Колонка `tariff_type` — рудимент, в pre-#08.3 скрывается из форм, в pre-#08.4 удаляется полностью.

Бриф pre-#08.3 составлен — 3 задачи: заголовки Add/Edit (6 сущностей), Detail charges/payments, скрытие тумблера.

### §1.2. Sonnet pre-#08.3 — три провалившихся пути и компромисс

Исходная архитектура (два form view на сущность, переопределение Edit через `LINKTOROW`) развалилась на блокере: системный Add «+» в deck view перехватился новой `*_Form_Edit`. Системный Add в текущей версии редактора — **read-only**, Target не редактируется.

Пробовали по очереди:
1. `LINKTOROW` + переопределение Edit, скрытие системного Edit — Add перехватился.
2. Action `App: open a form to add a new row` с явным Target — поле Target в текущей версии редактора **отсутствует**.
3. Переименование `*_Form_Edit` → `z_*_Form_Edit` — AppSheet кэширует выбор Add по внутреннему ID, не по имени. Не помогло.

Владелец трижды настоял на простоте — финальное решение: **VC `Section_Header` с универсальным заголовком «Создать/редактировать …»** на единственной системной form view. Sonnet нашёл паттерн в SKILL'е (строка 122 со сломанной ссылкой «See Section L»), вывел реализацию сам.

**Зафиксировано:**
- `appsheet-setup-SKILL.md` v2.16: новые Q.16 (Form header via Section_Header VC), Q.17 (Add/Edit single form view limitation), починка строки 122.
- `SYSTEM.md` §8.4: ограничение AppSheet принято как платформенное.

### §1.3. Два инцидента дисциплины SubGeneral

Зафиксированы в `REGULATIONS.md` v0.3 §1.7:

**Инцидент 7.** SubGeneral составил pre-#08.3 без `grep`'а SKILL'а. Решение для Add (тип action `App: open a form to add a new row`) лежало в строке 132 SKILL'а с момента pre-#06. Цена ~2 часа на эксперименты с переименованиями.

**Инцидент 8.** SubGeneral после Инцидента 7 прогрепил SKILL по словам `form, action, view`, но при смене постановки задачи на «универсальный заголовок» **не повторил grep** по `header, title`. Решение через VC Section_Header лежало в строке 122 SKILL'а. Цена ~30 минут + три гипотезы (race condition, откат, «нет решения») в разговоре с владельцем.

**Регламентное следствие:** добавлена `REGULATIONS.md` §7 «Дисциплина SubGeneral» (4 правила). Главное (§7.1): grep SKILL'а делается **до** составления брифа, и **повторяется** при каждой смене постановки внутри сессии.

### §1.5. Ресёрч встроенного фильтра AppSheet (вечер 10.05)

Совместно с владельцем (по скриншотам) проверено на view «ученики»:
- Двумерная фильтрация работает (на скрине одновременно `класс = 7 класс` И `дата начала обучения ≥ 3/1/2026`).
- Ref-колонки фильтруются.
- Диапазоны дат поддерживаются (>=, <=).
- Все колонки таблицы доступны без явной настройки.

**Открытие:** в проекте **нет прямых view** на charges и payments. Точка входа — `charges_filtered` и `payments_filtered`. То есть «удалить `*_filtered`» = заменить на новые обычные deck view со встроенным фильтром.

**Расходы (`expenses`)** — обычный view с встроенным фильтром, `expenses_filtered` параллельный рудимент к удалению.
