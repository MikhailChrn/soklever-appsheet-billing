**Дата составления:** 10.05.2026

**Кому:** Sonnet (без Adaptive thinking)
**Версия проекта на входе:** определить на старте (на 10.05 — 1.000489).
**Базовый бэкап:** v12.2 + результаты pre-#08.1 + результаты #08 (если #08 уже закрыт к моменту запуска этого брифинга — учесть его правки).

**Контекст:** UX-патч по результатам ревью владельца от 10.05.2026. Три однотипные проблемы (заголовки форм при редактировании показывают «Новый …» вместо «Редактирование …») и две сопутствующие (отсутствует Detail+Edit на карточках начислений/платежей; в форме тарифа виден рудиментарный тумблер «регулярный/единоразовый»).

**Очерёдность:** этот брифинг передаётся Sonnet'у **после** закрытия брифинга #08 (массовая замена на летний тариф, дедлайн 27.05). Pre-#08.3 — без жёсткого дедлайна.

**Принципиальное архитектурное решение владельца от 10.05.2026:**
> При добавлении новой записи без подзаголовка «Новый …» неудобно. Операционный директор работает в детской среде с постоянными отвлечениями на 15+ секунд — открыл форму, отвлёкся, вернулся, не помнит контекст. Поэтому: подзаголовок «Новый …» при добавлении остаётся, при редактировании — отдельный «Редактирование …».

Реализация — через два form view на сущность (Add — системный `*_Form`, Edit — новый `*_Form_Edit`) и переопределение action «редактировать» через `LINKTOROW`. Паттерн уже отработан в pre-#08.1 (см. `sonnet-skills/appsheet-setup-SKILL.md` §LINKTOROW).

## §4. Задачи

### §4.1. Универсальный заголовок формы — 6 сущностей

**🔄 Раздел переписан 10.05.2026** после провала исходной архитектуры (см. §4.1.0 «История узла»). Финальное решение — VC `Section_Header` с универсальным текстом «Создать/редактировать …». Применимо к Add и Edit одновременно.

#### §4.1.0. История узла (зафиксировано как ограничение AppSheet)

Исходная задумка — два form view на сущность (`*_Form` для Add, `*_Form_Edit` для Edit) с разными Display name. Edit развязывался через `LINKTOROW`. Add должен был остаться системным.

**Что не сработало (3 пути):**
1. **`LINKTOROW` + системный Edit скрыт.** Системный Add перехватился новой `*_Form_Edit` — «+» в deck view стал открывать «Редактирование» вместо «Новый».
2. **Action `App: open a form to add a new row` с явным Target.** В текущей версии редактора (май 2026) поле выбора целевой form view отсутствует.
3. **Переименование `*_Form_Edit` → `z_*_Form_Edit`.** AppSheet кэширует выбор Add по внутреннему ID, не по имени. После rename «+» продолжил открывать ту же view (с новым именем в шапке).

**Зафиксировано в SKILL'е:** `sonnet-skills/appsheet-setup-SKILL.md` Q.17 «Add/Edit form routing — single form view limitation».

**Финальное решение:** одна form view на сущность (системная `*_Form`). Внутри неё — VC типа `Show` категории `Section_Header`, поставленная **первой** в Column order. Display name VC = «Создать/редактировать …» — рендерится как заголовок и при Add (поля пустые), и при Edit (поля заполнены). Ограничение принято как платформенное.

#### §4.1.1. Архитектура (одинаковая для всех 6 сущностей)

| Артефакт | Описание |
|---|---|
| VC `_form_header_<table>` | Type: Show, Category: Section_Header, App formula: `""`, Display name: «Создать/редактировать <сущность>» |
| Column order Manual в `*_Form` | VC `_form_header_<table>` стоит **первой**, далее обычные поля в нужном порядке |
| Системный Edit | НЕ трогаем (остаётся в дефолте) |
| Системный Add | НЕ трогаем (остаётся в дефолте) |

**Никаких `*_Form_Edit`. Никаких пользовательских `*__edit` actions. Никаких переименований.** Если в ходе предыдущих попыток что-то из этого было создано — удалить, вернуть системный Edit в дефолт.

#### §4.1.4. Что делать с редактированием «Авто»-начислений

Исходно бриф предусматривал ограничение: «Авто»-charges не должны быть редактируемыми (ломает «твёрдую землю»). Это ограничение реализовывалось через `Show_If` на пользовательском `charges__edit` action. **Раз пользовательский action не создаётся** (одна универсальная form view, системный Edit не трогаем) — ограничение в pre-#08.3 **не реализуется**.

Перенесено в открытые хвосты `SYSTEM.md` §8.4: «Запрет Edit для charges с charge_type = Авто — реализовать через Show_If на системном Edit, если AppSheet это позволяет; иначе — через политику, не через UI». Будет рассмотрено отдельно после pre-#08.3.

**В отчёт:** скриншот формы любой из 6 сущностей с заголовком «Создать/редактировать …» — для подтверждения, что VC отрендерилась как заголовок секции.

### §4.2. Карточки charges и payments — Detail view + Edit

**Сейчас** при просмотре карточки начисления и платежа: нет заголовка фрейма, нет кнопки «редактировать» (см. ревью владельца 10.05.2026, п.4).

**Гипотеза:** системные `charges_Detail` / `payments_Detail` существуют и используются (карточки же открываются), но настроены минимально. Проверить:

| # | Шаг | Где |
|---|-----|-----|
| 4.2.1 | Найти `charges_Detail` и `payments_Detail` в Views (обычно system-generated, могут быть скрыты в фильтре «System views») | Views |
| 4.2.2 | Если их нет — создать вручную (тип Detail, For this data — соответствующая таблица, Position: ref) | Views → + |
| 4.2.3 | В каждом — выставить **Header column** (поле, отображаемое в шапке карточки), пример: для charges — `_ComputedName` (если есть) или `description`, или `period`+`student_id`. Для payments — аналогично. Точное поле выбирается на основании того, что максимально информативно для оператора. Если есть сомнения — поднять стратегу через владельца, не догадываться. | Detail view → Header column |
| 4.2.4 | В каждом — убедиться, что action `charges__edit` / `payments__edit` (созданные в §4.1.3.1) видны на карточке (Position: Prominent в action'е и не скрыты Show_If view) | Behavior + View |

**Особенность для charges:** action `charges__edit` показывается только при `[charge_type] = "Ручное"` (§4.1.3.2). Для «Авто»-начислений кнопка скрыта — это ожидаемо.

**В отчёт:** два скриншота — карточка `charges` (с заголовком и кнопкой «редактировать», если запись «Ручное»; без кнопки, если «Авто») и карточка `payments` (с заголовком и кнопкой «редактировать»).

### §4.3. Скрыть тумблер `tariff_type` в формах тарифа

**Контекст:** в карточке «Новый тариф» / «Редактирование тарифа» показан Enum-тумблер «Регулярный / Единоразовый». **Архитектурное решение от 10.05.2026** (зафиксировано в `SYSTEM.md` §4.3, `GLOSSARY.md` §2.9): все тарифы по определению регулярные. Разовые сборы реализуются через `charges[charge_type] = "Ручное"` (точечное начисление), а не через отдельный «единоразовый тариф». Значение «Единоразовый» никогда не использовалось — рудимент.

**В этом брифинге** (pre-#08.3) — только **скрытие из форм** + Initial value, чтобы тумблер не маячил перед оператором.

**Полное удаление** колонки `tariff_type` (правка фильтра в `bot_close_period`, удаление колонки в Sheets, чистка Enum-значений) — задача **pre-#08.4**. Поэтому в pre-#08.3 колонку **не удаляем**, фильтр бота **не трогаем**.

| # | Шаг | Где |
|---|-----|-----|
| 4.3.1 | На колонке `tariff_type` (Data → tariffs → tariff_type) — Initial value = `"Регулярный"` | Data → Columns |
| 4.3.2 | В `tariffs_Form` (Column order Manual) — убрать `tariff_type` из списка показываемых полей (Remove). | Views → tariffs_Form → Column order |
| 4.3.3 | Аналогично для `tariffs_Form_Edit` (Column order Manual → Remove). | Views → tariffs_Form_Edit → Column order |
| 4.3.4 | Show? у колонки в Data → Columns оставить **true** — она ещё нужна в Detail view тарифа и в фильтре `bot_close_period`. После pre-#08.4 колонка исчезнет вместе с Detail-полем. | Data → Columns |
| 4.3.5 | Проверить: при добавлении нового тарифа — поля `название`, `сумма`, `комментарий` (без тумблера). Сохранение даёт `tariff_type = "Регулярный"`. | Тестирование |

**Sanity check:** после изменений Initial value `bot_close_period` должен продолжать работать как раньше — фильтр `[tariff_type] = "Регулярный"` истинен для всех тарифов (включая новые, где Initial value подставит «Регулярный»). Для этого брифинга — **не запускать бот**, только проверить через скриншот, что `tariff_type` нового тестового тарифа сохранён как «Регулярный» (видно в Sheets).

**В отчёт:** скриншот формы «Новый тариф» без тумблера + строка нового тестового тарифа в Sheets с `tariff_type = "Регулярный"`.

## §5. Архитектурный контекст (справочно, НЕ ПРАВИТЬ)

### §5.1. Почему два form view вместо одного

AppSheet берёт **Display name** form view как статический подзаголовок шапки фрейма. Один form view не может показать «Новый …» при Add и «Редактирование …» при Edit — выражения в Display name не разрешены или не разрешают `LINKTOROW`-режим как условие. Поэтому идиома: **два form view, переключение через action**.

Шапка фрейма (top, рядом с крестиком) — это **Header column** записи (например, имя записи). Она и так показывает «5 класс, база» при редактировании. Подзаголовок («Новый тариф») — это Display name. Их два.

### §5.2. Почему системный Edit, а не отдельный Slice

В pre-#08.1 для выбытия делался Slice `students_withdrawal` с Update mode = Updates only. Там это было нужно: форма содержала **единственное** служебное поле `withdrawal_date`, не являющееся ключом, и Save должен был именно **обновить** существующего ученика, а не создать нового.

Здесь иначе: пользователь редактирует запись с **существующим ключом**. AppSheet это распознаёт сам — Save обновит запись, кнопки «+» в Edit-форме не появится. Slice — лишняя сущность.

### §5.3. Почему Edit для «Авто»-начислений запрещён

См. `SYSTEM.md §4.5`, `SYSTEM.md §2.2`, `GLOSSARY.md §1.2` (твёрдая земля). Закрытый период — твёрдая земля, цифры зафиксированы. Корректировка задним числом ломает базовую гарантию. Корректировки реализуются добавлением «Ручное» с минусом — это не отменяет «твёрдую землю», а добавляет наблюдаемое движение в живой зоне.

### §5.4. Почему `tariff_type` не удаляем

Бот `bot_close_period` использует `tariffs[tariff_type] = "Регулярный"` как фильтр в ForEach по `student_tariffs` (создаются charges только по регулярным тарифам). См. `ARCHITECTURE.md §4`. Удаление колонки = поломка бота.

Скрытие из форм — безопасно, потому что Initial value покрывает все новые тарифы.
