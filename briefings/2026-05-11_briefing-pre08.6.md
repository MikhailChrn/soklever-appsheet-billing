### §4.2. Защита от создания charges в закрытом периоде

**Контекст:** в pre-#08.5 §4.4 на системном Edit `charges` поставлен Show_If `[period] = INDEX(settings[current_period], 1)` — Edit виден только для charges текущего открытого периода. Это закрыло **редактирование** закрытого. Но **создание** новой charge в закрытом периоде через «+ начисление» в карточке ученика — пока разрешено: пользователь выбирает `period` вручную в Ref-dropdown.

**Дыра:** создав новую строку (не отредактировав старую), можно обойти твёрдую землю. Пример: ADMIN кликает «+ начисление», выбирает period `202604` (закрытый), вводит сумму — charge создаётся в закрытом периоде, ломая фиксацию закрытия по Р-7.

**Цель:** на колонке `charges[period]` ограничить выбор в Ref-dropdown списком разрешённых периодов — **текущий открытый + 3 следующих** (±3 — окно, принятое в проекте, см. `sonnet-skills/appsheet-setup-SKILL.md` строка 547; «-3» отключаем, т.к. ретро запрещено §4.4 pre-#08.5).

Дополнительно — Initial value: автоматически подставлять текущий период, чтобы пользователь по умолчанию не выбирал ничего.

#### §4.2.1. Контекст по AppSheet

`Valid_if` на Ref-колонке в AppSheet **фильтрует dropdown** (не только валидирует запись), если возвращает список ключей таблицы-источника. Подтверждено в проекте 03.05.2026 (`REGULATIONS.md` §1.7, Инцидент по Valid_if). Это и есть нужный нам механизм — список разрешённых периодов сужает выбор в UI **и** валидирует данные.

Паттерн `INDEX(settings[current_period], 1)` для извлечения скаляра — принят в проекте (`sonnet-skills/appsheet-setup-SKILL.md`, обращения в `bot_close_period`, VC `settings[current_period]`).

#### §4.2.2. Настройка колонки `charges[period]`

| # | Параметр | Значение |
|---|----------|----------|
| 4.2.2.1 | **Initial value** | `INDEX(settings[current_period], 1)` |
| 4.2.2.2 | **Valid_if** | `SELECT(ref_periods[period], AND(NUMBER([period]) >= NUMBER(INDEX(settings[current_period], 1)), NUMBER([period]) <= NUMBER(INDEX(settings[current_period], 1)) + 3))` |

**Объяснение формулы Valid_if:**
- `SELECT(ref_periods[period], <условие>)` — возвращает список ключей `ref_periods`, удовлетворяющих условию. AppSheet видит, что это список ключей Ref-таблицы → фильтрует dropdown.
- `NUMBER([period]) >= NUMBER(INDEX(settings[current_period], 1))` — нижняя граница: текущий открытый период.
- `NUMBER([period]) <= NUMBER(INDEX(settings[current_period], 1)) + 3` — верхняя граница: текущий + 3 вперёд.
- Арифметика на числовом коде `YYYYMM`: `202612 + 1 = 202613`, что не существует в `ref_periods`. SELECT естественно отсекает несуществующие коды — на границе года список будет короче. Поведение принято в SKILL §547.

#### §4.2.3. Расположение `period` в форме

Не скрывать. Пользователь должен видеть, в какой период идёт начисление (наглядность). Защита держится на Valid_if, не на сокрытии поля.

#### §4.2.4. Тесты

| # | Сценарий | Ожидаемое |
|---|----------|-----------|
| 4.2.4.1 | Открыть форму «+ начисление» в карточке ученика | `period` уже подставлен — текущий открытый период (например, `202606`) |
| 4.2.4.2 | Раскрыть dropdown `period` | В списке: `202606` (текущий) + до трёх следующих (`202607`, `202608`, `202609`). Закрытые периоды (`202605` и раньше) и далёкое будущее (`202610`+) — не доступны. |
| 4.2.4.3 | Попытаться сохранить charge с вручную проставленным закрытым периодом (если получится через Ref) | Save блокируется ошибкой валидации |
| 4.2.4.4 | Сохранить charge с текущим периодом | Успех |

**В отчёт:** скриншот dropdown (список периодов), скриншот формы с подставленным period, результаты тестов.
