## §1. Контекст и цель

### §1.1. Зачем

В мае ученики занимаются по **классным тарифам** (база + продлёнка). Летом школа переходит на **единый летний тариф** T010. Чтобы это сработало без ручного редактирования каждой привязки, нужен массовый скрипт.

**Бот `bot_close_period` создаёт начисления за СЛЕДУЮЩИЙ период** на основе действующих привязок. Поэтому замена должна отработать **до закрытия мая** — иначе майские charges будут уже созданы по новым привязкам, а нам нужны июньские.

См. `GLOSSARY.md §2.3` (предупреждение про «действующие в момент нажатия» привязки) и `SYSTEM.md §5.6` (полный летний цикл).

| # | Параметр | Значение |
|---|----------|----------|

**Process steps (последовательно, без Branch):**

| # | Step | Type | Что делает |
|---|------|------|-----------|
| 4.3.6.6 | Step 1 «delete tariffs» | Run a data action → `settings__delete_all_active_student_tariffs` | Удаляет все привязки активных учеников |
| 4.3.6.7 | Step 2 «create summer» | Run a data action → `settings__create_summer_tariffs_for_active` | Создаёт T010 для каждого активного |
| 4.3.6.8 | Step 3 «log» | Run a data action → Set row values на settings | `last_summer_apply_at = NOW()`, `last_summer_apply_message = CONCATENATE("Замена на летний тариф выполнена. Активных учеников: ", COUNT(SELECT(students[student_id], [status]="Активен")), ". Создано привязок: ", COUNT(SELECT(student_tariffs[assignment_id], [tariff_id]="T010")), ".")` |

**⚠️ Порядок шагов критичен.** Сначала delete, потом create, потом log. Если порядок переставить — старые привязки могут быть удалены после создания новых, или log снимет неправильное состояние.

**⚠️ Анти-каскад здесь не нужен** (в отличие от `bot_close_period`). Bot отработает один раз на одно изменение `last_summer_apply_trigger`. Повторное нажатие создаст ещё один триггер → ещё один прогон. Но **повторный прогон безопасен**: delete снова удалит уже изменённые привязки (включая T010), create снова создаст T010. Идемпотентно.

## §5. Архитектурный контекст (справочно, НЕ ПРАВИТЬ)

### §5.1. Почему именно «удалить всё + создать одно», а не «обновить»

`student_tariffs` — таблица связи многие-ко-многим. У одного ученика может быть 2-3 привязки (база + продлёнка). Update не решает задачу — нужно сначала вычистить все, потом создать одну. AppSheet не поддерживает «replace all by filter» одной операцией, поэтому делаем в два шага.

### §5.2. Почему через служебный триггер, а не Bot прямо на User Action

В AppSheet нет события «Bot on action click». Все Bot'ы запускаются на Data Change. Идиома (Q.2 SKILL'а): User Action пишет `NOW()` в служебную колонку `settings`, Bot слушает её. Это уже работающий паттерн в проекте — `bot_close_period` устроен так же.

### §5.3. Почему идемпотентно, а не с защитой

Скрипт **должен быть** запускаем повторно, потому что:
- Если новый ученик добавится после первого прогона — нужно прогнать ещё раз, чтобы он тоже получил T010.
- Если что-то сорвётся — нужно повторить.

«Защита от повторного нажатия» здесь не нужна (в отличие от `bot_close_period`, где повтор давал каскад начислений). Здесь повтор приводит к тому же состоянию: все активные → одна привязка к T010.

### §5.4. Почему не делаем возвратный скрипт

01 сентября при пересборке классов владелец вручную:
- Меняет `class_group` ученикам (новый учебный год — все на класс выше).
- Удаляет привязки T010.
- Создаёт классные привязки на основе нового состава.

Это разовая операция в году, делать её через скрипт — лишняя сложность. «Суровая правда без шума» (`GLOSSARY.md §1.3`).

### §5.5. Связь с дашбордом

После применения скрипта формула `current_period` и расчёты дашборда работают как обычно. **Изменится только** будущий результат закрытия мая: вместо классных charges будут T010-charges по 5 000 ₽ × количество активных учеников. Это **ожидаемо**.
