## §2. Архитектурные принципы

### §2.1. Модель сосудов (из v8)

У каждого ученика — «сосуд». Уровень сосуда = баланс (`debt`):
- Положительный = ученик должен.
- Нулевой = квит.
- Отрицательный = ученик переплатил.

Платежи и начисления **не привязаны друг к другу**. Деньги падают в общий сосуд ученика, начисления его наполняют.

### §2.2. Живая зона vs твёрдая земля (из v8)

**Живая зона** = текущий период (`current_period`). Здесь данные ещё могут меняться: новые платежи, разовые начисления, корректировки.

**Твёрдая земля** = всё, что строго раньше `current_period`. Не редактируется, замороженный итог. Используется для расчёта долгов.

После закрытия периода старый месяц автоматически становится твёрдой землёй. Без явного «закрытия» — сама смена `current_period` это делает.

**Семантическая заметка:** после закрытия периода **следующий период становится «текущим»** (`current_period` сдвигается на новый, только что открытый). То есть «текущий период» в системе ≠ «период, который мы только что закрыли». Это важно для понимания формул и текстов сообщений.

### §2.3. `tariff_type` vs `charge_type` — две разные оси (новое в v10)

В v9 §6.1 п.3 я ошибочно зафиксировал, что `tariffs[tariff_type]` — рудимент. На самом деле это **рабочая колонка**, использующаяся в фильтрах. Поправка в v10:

**`tariffs[tariff_type]`** — тип регулярности тарифа:
- Сейчас все 9 тарифов имеют `tariff_type = "Регулярный"`. Других значений в данных нет.
- В Enum возможны и другие значения — например, в будущем `"Разовый"` для T009 «Летний взнос» (см. §8 хвост).
- Используется в фильтрах:
  - Bot ForEach (`step_no_create_charges`): отбирает регулярные тарифы для генерации charges.
  - `settings[current_period]`: отбирает period из charges по регулярным тарифам.

**`charges[charge_type]`** — источник начисления:
- Только два значения после #07.3: `"Авто"` (создаются ботом при закрытии периода) и `"Ручное"` (создаются через форму разового начисления, карточка ученика).
- `"Корректировка"` удалена в #07.3. «Корректировки» теперь — «Ручное» с отрицательной суммой.

Это **две разные оси.** При написании формул не путать — особенно потому что в начале #07 тестовая ошибка возникла именно из-за смешивания (Branch фильтровал `charges[charge_type] = "Регулярный"`, такого значения в `charges` нет в принципе).

### §2.4. Суровая правда без шума (из v8)

Система показывает факты, не интерпретирует. Должен 12 500? Это число — точка. Не «должен за март столько, за апрель столько» — система не различает по месяцам.

**Простота важнее «гибкости на будущее»** — если фича не нужна сейчас, не закладывать «на всякий случай».

### §2.5. Одно место правды (из v8)

Каждое значение хранится в одном месте. `tariffs[amount]` — единственный источник суммы тарифа. `student_tariffs[tariff_amount]` — VC-отображение, не дубликат.

### §2.6. Работать без владельца (из v8)

Любой человек, открывший проект через год без объяснений, должен понимать архитектуру по самому проекту. Логика — в редакторе AppSheet (видно), не в скрытом Apps Script.

## §4. Все формулы дословно

### §4.1. Дашборд — Живая зона (открытый период)

**`settings[dash_открытый_поступления]`** (Number, VC):
```
SUM(SELECT(payments[amount],
  AND(
    YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
    MONTH([payment_date]) = NUMBER(RIGHT(ANY(settings[current_period]), 2))
  )
))
```

**`settings[dash_открытый_расходы]`** (Number, VC):
```
SUM(SELECT(expenses[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**`settings[dash_открытый_начислено]`** (Number, VC):
```
SUM(SELECT(charges[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**Архитектурное наблюдение:** три формулы используют **разные** способы определить «текущий период»:
- Поступления — по `payments[payment_date]` через `YEAR()`+`MONTH()`. Потому что у `payments` нет `period`.
- Расходы и начисления — по `[period] = current_period`. Потому что у `expenses` и `charges` `period` есть.

Это **намеренно**, не баг. Платёж бэк-датой за март попадает в мартовские поступления, не в апрельские.

### §4.2. Дашборд — Долги и Твёрдая земля

**`settings[dash_долги_количество]`** (Number, VC):
```
COUNT(SELECT(families[family_id], [family_overdue_debt] > 0))
```

⚠️ **Открытый вопрос v10:** в #07.3 в NO-сообщении бота это поле дало значение `1`, хотя плитка «Долги» на дашборде в то же время показывала `3`. Гипотеза: формула считается на pre-commit состоянии транзакции бота — VC ещё не пересчитались после ForEach. После транзакции значения сходятся. См. §7 хвост.

**`settings[dash_долги_сумма]`** (Number, VC):
```
SUM(SELECT(families[family_overdue_debt], [family_overdue_debt] > 0))
```

**`settings[dash_земля_получено]`** (Number, VC):
```
SUM(SELECT(payments[amount],
  AND(
    [payment_date] >= ANY(settings[academic_year_start]),
    OR(
      YEAR([payment_date]) < NUMBER(LEFT(ANY(settings[current_period]), 4)),
      AND(
        YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
        MONTH([payment_date]) < NUMBER(RIGHT(ANY(settings[current_period]), 2))
      )
    )
  )
))
```

**`settings[dash_земля_израсходовано]`** (Number, VC):
```
SUM(SELECT(expenses[amount],
  AND(

### §4.4. Дебиторская задолженность (просрочка)

  SUM(SELECT(charges[amount],
    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))
    )
  ))
  - SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id]))
  > 0,

    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))

  0

```

Эквивалент `MAX(0, ...)`, написанный длинно через IFS. Выражение продублировано — **при правке менять в обоих местах**. Кандидат на упрощение через `MAX(0, ...)`.

### §5.2. Финальная архитектура (UPD после #07.3)

**Платформа:** AppSheet Bot, триггерится через служебную колонку `settings[last_close_trigger]`.

**Цепочка:**

1. Пользователь на view «скрипты» нажимает кнопку «Закрыть период».
2. Срабатывает Action `settings__close_period_trigger` (`Data: execute an action on a set of rows`).

## §7. Открытые вопросы

### §7.1. Текст NO-сообщения

Текущая формула (см. §5.8) использует `ANY(settings[current_period])` для номера периода и говорит «Период X закрыт». Но после успешного закрытия `current_period` — это **новый открытый** период, а не закрытый. Текст некорректен.

Варианты переформулировки (на выбор владельца):

- **Без указания закрытости:** «Создано начислений за период {current_period}: N. Должников: T.»
- **С указанием обеих сторон:** «Закрытие выполнено. Открытый период: {current_period}. Создано: N. Должников: T.»
- **Через предыдущий период (хелпер `dash_пред_период` уже есть):** «Период {dash_пред_период} закрыт. Открытый: {current_period}. Создано: N. Должников: T.»

Решение откладывается до согласования с владельцем. Мини-патч на одну формулу.

### §7.2. `dash_долги_количество` — расхождение pre/post-commit

В #07.3 в NO-сообщении значение `dash_долги_количество` было `1`, в то время как плитка «Долги» на дашборде показывала `3`.

Что делать — три пути:
1. Принять как есть, **убрать «Должников» из NO-сообщения** (на дашборде есть актуальное значение).
2. Понять механику пересчёта VC внутри bot transaction (возможно, поможет другая последовательность шагов).
3. Использовать прямой подсчёт в формуле сообщения, без обращения к `dash_долги_количество`.

Не блокер. Решать вместе с §7.1 в следующем мини-патче.

## §9. Открытые «руки» — карта подводных камней AppSheet

Из v9 + наблюдения #10:

1. **Ручной триггер Bot** не существует как отдельный тип event. Идиоматично — служебная колонка-триггер + Data Change Event.
2. **Синтаксис `[_THISROW_BEFORE]` / `[_THISROW_AFTER]`** работает только в Bot Event Condition.
3. **Синтаксис `<<...>>` в Confirmation Message не работает.** Использовать `CONCATENATE`.
4. **Notification body — `CONCATENATE` работает.** Подтверждено в #07.3 (старый бот использовал такой синтаксис).
5. **«Send a notification»** шлёт push на устройство (требует AppSheet native app + системные пуши включены), а не сообщение в inbox внутри приложения. Если push не настроены — сообщение не доставляется. **Альтернатива:** запись в settings + плитка на дашборде (это и сделали в #07.3).
6. **`Data: execute an action on a set of rows` не работает в multi-record view (Deck/Table)** как глобальная кнопка — показывается на каждой строке. Обход: однострочная таблица + отдельный view-фрейм.
7. **`CONTEXT("View")`** возвращает имя текущего view. Использовать в Show_If.
8. **Same-table reference action** разрешён.
9. **KEY-колонку нельзя скрыть на writable-таблице.** Обход: read-only Slice (см. хвост в §5.9).
10. **VC внутри Bot transaction:** значения пересчитываются лениво. Если в шаге `Set row values` обращаться к VC, можно получить pre-commit состояние. См. §7.2.
