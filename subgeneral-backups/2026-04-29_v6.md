### 1.2. Стиль общения с владельцем

**Главное правило:** короткие ответы, без простыней. Владелец читает наискосок, выхватывает выводы.

**Из этого следует:**
- Выводы сверху, детали ниже. Не наоборот.
- Без формул и кода в живом обсуждении. Формулы — только в брифингах для Sonnet.
- Метафоры и образы — рабочий язык.
- Не убегать вперёд, дожимать каждый вопрос до конца.
- Не выкатывать монологи. Спрашивать и предлагать. Дать владельцу решать.
- Не оправдываться при поправках. Воспринимать и корректировать.

**Когда уместны формулы:** только когда владелец сам их просит, или в брифинге для Sonnet.

**Сигнал перегруза:** если владелец говорит «много написал», «не понял», «можно сжать» — это команда сократить ответ в 2–3 раза. Не пытаться объяснить ту же мысль другими словами — выкинуть половину пунктов.

- **Primary меню** — нижняя панель навигации в AppSheet на телефоне.
- **Menu** — выпадающее боковое меню (гамбургер).

## 2. Модель мира

### 2.1. Метафора двух сосудов

**Это центральная ментальная модель проекта.** Все остальные решения опираются на неё.

Платёж не гасит конкретное начисление. Платёж и начисление работают как **два сообщающихся сосуда**:
- В сосуд «Начислено» льются charges (всё, что школа выставила ученику).
- В сосуд «Оплачено» льются payments (всё, что родители заплатили).
- Разница между сосудами = долг ученика.

**Что из этого следует:**
- У платежа нет «своего периода» — он просто меняет уровень в общем сосуде.
- Долг — динамическая величина, не снимок во времени.
- «Закрыть период» не означает «зафиксировать какое-то состояние сосудов». Сосуды плещутся всегда.

### 2.2. Метафора «твёрдая земля + живая зона»

**Концептуальное разделение времени в системе:**
- **Твёрдая земля** — закрытые периоды. История. Цифры зафиксированы (но не как снимок — а как невозможность нового движения в этих периодах).
- **Живая зона** — текущий открытый период. Тут сосуды плещутся, начисления и платежи продолжают двигаться.

**Что из этого следует:**
- Дашборд разделён на блоки по этим зонам.
- Должник = тот, кто не доплатил за твёрдую землю. За живую зону долгом ничего не считается.

### 2.3. Принцип «суровой правды жизни без лишнего шума»

Система разрабатывается простой. Не закладываем механики, которые ничего не привносят:
- Нет историй тарифов.
- Нет нераспределённых платежей.
- Нет специальных снимков состояния.
- Нет автоматического прогнозирования.

Когда возникает соблазн усложнить — спрашиваем «что это даёт прямо сейчас?». Если ответ «гибкость на потом» — отклоняем. На промышленном стеке (если будет) модель будет переписана с нуля.

### 2.4. Двухконтурный учёт (на вырост)

При переходе на промышленный стек вероятно появится двухконтурный бухучёт. Это **не реализуем сейчас**, но это объясняет, почему `expenses[period]` остаётся (см. Р-2).

## 3. Архитектурные решения (ADR)

### Р-1. `payments[period]` убрано ✅

Удалено поле `period` из таблицы `payments`. Платёж не привязан к периоду в модели сосудов.
**Реализовано в брифинге #1 (28.04).** Slice `payments_filtered`, дашбордные блоки переведены на `payment_date`.

### Р-2. `expenses[period]` оставляем как есть

Без признака «регулярный/разовый». Двухконтурный учёт на промстеке потребует разделения «дата факта» и «период отнесения». Усложнение через `is_periodic` не оправдано на этом этапе.

### Р-3. `charges[period]` без изменений

Семантика начисления привязана к периоду. Не обсуждается.

### Р-4. `payment_date` редактируется оператором, default = TODAY()

Оператор может проставить дату платежа задним числом для стыков месяцев. По умолчанию — сегодня. В 95%+ случаев оператор не трогает дату.

### Р-5. `debt` — динамический, без истории

Долг считается на лету. Не хранится, не фиксируется снимками. Список должников — view в AppSheet, собирается из текущих данных.

**Семантика `opening_balance`:** положительное значение = предоплата ученика на момент старта системы; отрицательное = долг на момент старта. В формулах долга `opening_balance` **вычитается** (см. Р-6 формулу).

### Р-6. Должник = `overdue_debt > 0` по периодам **строго раньше активного** ✅

**Жёстко зафиксировать.** Это решение уже выветривалось между ветками — не дать выветриться снова.

Если ученику начислили апрель и апрель ещё идёт — это не долг, это нормальная неоплата текущего периода. Должником ученик становится, когда период закончился, а долг по нему остался.

**Концептуально:** должник = тот, кто не доплатил за «твёрдую землю». Платежи в формуле должника учитываются ВСЕ — потому что сосуд общий.

**Активный период** определяется через последнее регулярное начисление в `charges` — VC `settings[current_period]`.

**Боевая формула `students[overdue_debt]` (реализовано в брифинге #1):**
```
IFS(
  SUM(SELECT(charges[amount], AND([student_id]=[_THISROW].[student_id], NUMBER([period]) < NUMBER(ANY(settings[current_period])))))
  - SUM(SELECT(payments[amount], [student_id]=[_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id]=[_THISROW].[student_id]))
  > 0,
  [та же формула без сравнения],
  TRUE,
  0
)
```
`families[family_overdue_debt]` = сумма `overdue_debt` по семье.

### Р-7. Закрытие периода — в последних числах закрываемого месяца

Апрель закрываем 28–30 апреля, не 5–10 мая. Большая часть платежей за апрель попадает в систему до конца апреля. Хвостовые случаи на стыке решаются через Р-4.

### Р-8. Тарифы и инфляция — переписываем `amount` в справочнике

При повышении тарифа просто переписываем `amount` в `tariffs`. История справочника не нужна. Прошлые начисления не пострадают (в `charges` хранится фактическая сумма на момент создания).

### Р-9. Дашборд: «твёрдая земля + живая зона»

Дашборд для руководителя школы. Отвечает на 4 вопроса:
1. Сколько денег в казне сейчас?
2. Сколько ещё придёт в этом месяце?
3. Сколько нам должны?
4. Что с расходами в этом месяце?

**Структура — три блока в порядке от ближнего к дальнему:**

**Блок 1. Текущий период (живая зона):** поступления и расходы за текущий месяц по `payment_date`/`expense_date`, ожидаемые поступления.
**Блок 2. Долги:** количество должников и общий долг (по Р-6).
**Блок 3. Накоплено за учебный год (твёрдая земля):** поступления, расходы, свободные средства с 1 сентября до конца последнего закрытого периода.

Прогнозирование не закладываем.

**Статус:** архитектура зафиксирована. Реализация — задача ветки v7 (брифинг #3).

### Р-10. Закрытие учебного года — обычная финансовая процедура

Никакой специальной механики «закрытия года» в коде. В августе: снимаем физический остаток денег → проводим как `expenses` → накопительные итоги обнуляются. 1 сентября — нулевая твёрдая земля.

### Р-11. Летний цикл

- **31 мая** — закрытие мая.
- **1 июня** — массовая смена тарифа на «Летний» (через скрипт).
- **Июнь–август** — обычные циклы закрытия.
- **Конец августа** — закрытие августа + ручной перенос остатка по Р-10.
- **1 сентября** — **ручная** пересборка: классы, состав учеников, индивидуальные тарифы.
- **30 сентября** — закрытие сентября.

### Р-12. Цветовая индикация долгов ✅

В карточке ученика и семьи долги окрашиваются:
- **Текущая задолженность** > 0 → синий.
- **Дебиторская задолженность** > 0 → красный.
- Любой долг < 0 (переплата) → зелёный.
- 0 → нейтральный (без правила).

Реализовано через 8 Format Rules в брифинге #2 (29.04).

### Р-13. Inline-блоки в карточке ученика — таблицы, не deck ✅

В карточке ученика «текущие платежи» и «текущие начисления» отображаются как **table**, не deck. Принято как боевое решение — компактно, нужные колонки видны. Не считается хвостом.

## 5. Best Practices

### Технические (AppSheet/Sheets)

**BP-1.** Sheets — источник истины для владельца. AppSheet — операторский UI.
**BP-2.** `settings` — служебная таблица (одна строка) для текущих параметров.
**BP-5.** Чтение `settings`: `ANY(settings[column])` или `INDEX(settings[column], 1)`.
**BP-6.** Фильтр Slice «пусто = все»: `OR(ISBLANK(ANY(settings[field])), [field] = ANY(settings[field]))`.
**BP-7.** MAX не работает на Enum/Text. Использовать `INDEX(SORT(SELECT(...), TRUE), 1)`.
**BP-8.** TEXT() принимает только Date, не Number.
**BP-15.** FK через Ref. Сравнение `[a] = [b]` без TEXT-обёрток.
**BP-17.** Slice пересоздавать, если формула не сохраняется (баг кеширования). См. также BP-28.
**BP-18.** Update mode Slice = `ADDS_AND_UPDATES` для появления «+». Для READ-ONLY view — `READ_ONLY`.
**BP-19.** Заголовок формы через VC: type `Show`, category `Section_Header`, App formula `""`, Display name = текст, Show? = `true` **без условий** (см. BP-23). В Manual order — первой и только колонки своей таблицы.
**BP-20.** Сравнение Enum-периода (YYYYMM) с динамической датой: `NUMBER([period]) >= (YEAR(TODAY() - N) * 100 + MONTH(TODAY() - N))`.
**BP-22.** В AppSheet нет VALUE(). Для парсинга Text/Enum в число — `NUMBER()`.
**BP-23.** Заголовок-VC: Show? = `true` **без условий**. Условные Show? (например, `CONTEXT("ViewType") = "Form"`) ведут к разному поведению при входе через action (LINKTOFORM) и через штатную «+»-кнопку — заголовок появляется только в одном из сценариев. Подтверждено в брифинге #2.
**BP-24.** В AppSheet `MAX()` принимает только 1 аргумент. Для клипа к нулю использовать `IFS(value > 0, value, TRUE, 0)`.
**BP-26.** Inline-Related-блок через slice. Чтобы Related-колонка в Detail-карточке рендерилась через slice (с фильтрацией): Data → таблица → колонка `Related X` → Type Details → **Referenced table name** → выбрать `slice_name (slice)`. Без этого блок берёт всю таблицу.
**BP-27.** Deck требует Label у источника. Если у source-таблицы/slice не отмечена ни одна колонка как Label — deck молча падает в table-рендер. Решение: Data → Columns → таблица → поставить галку Label у одной колонки (рекомендация — у поля типа `description`).
**BP-28.** Slice удаляет VC при сохранении, если у slice ограниченный набор Slice Columns. Порядок: (1) в slice временно поставить Slice Columns = `*`, (2) создать VC в root-таблице, (3) Regenerate Schema у таблицы, (4) только потом вернуть явный список колонок в slice. Уточнение к BP-17.
**BP-29.** Format Rules — путь: UX (иконка телефона) → подменю → **Format Rules** (не через шестерёнку рядом с Views — там Settings). Создание: `+` → For this data → условие → Format these columns → Visual Format (Text color / Highlight color).
**BP-30.** KEY-колонку нельзя скрыть через Show? = false на записываемой таблице. Для скрытия в форме — использовать Column order Manual в форме и не добавлять KEY-колонку в список.

### Процессные

**BP-25.** Решения, выветренные между ветками, фиксировать жёстче. Если на старте проекта решение было принято, а через 2–3 итерации sub-general его нет в бэкапе — это признак потери. Возвращать в бэкап с пометкой «жёстко зафиксировать».

### О-1. Что именно делает процедура закрытия периода?

Развилка:
- (а) Генерирует начисления на следующий месяц из `student_tariffs`.
- (б) Создаёт неизменяемый снимок должников.
- (в) Что-то ещё.

**Когда решать:** до реализации Apps Script.

### О-2. Судьба таблицы `debts` (в эталоне Sheets)
