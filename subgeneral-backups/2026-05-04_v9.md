## §2. Архитектурные принципы (из v8)

### §2.1. Модель сосудов

У каждого ученика — «сосуд». Уровень сосуда = баланс (`debt`):
- Положительный = ученик должен.
- Нулевой = квит.
- Отрицательный = ученик переплатил.

Платежи и начисления **не привязаны друг к другу**. Деньги падают в общий сосуд ученика, начисления его наполняют.

### §2.2. Живая зона vs твёрдая земля

**Живая зона** = текущий период (`current_period`). Здесь данные ещё могут меняться: новые платежи, разовые начисления, корректировки.

**Твёрдая земля** = всё, что строго раньше `current_period`. Не редактируется, замороженный итог. Используется для расчёта долгов.

После закрытия периода старый месяц автоматически становится твёрдой землёй. Без явного «закрытия» — сама смена `current_period` это делает.

### §2.3. Суровая правда без шума

Система показывает факты, не интерпретирует. Должен 12 500? Это число — точка. Не «должен за март столько, за апрель столько» — система не различает по месяцам.

**Простота важнее «гибкости на будущее»** — если фича не нужна сейчас, не закладывать «на всякий случай».

### §2.4. Одно место правды

Каждое значение хранится в одном месте. `tariffs[amount]` — единственный источник суммы тарифа. `student_tariffs[tariff_amount]` — VC-отображение, не дубликат.

### §2.5. Работать без владельца

Любой человек, открывший проект через год без объяснений, должен понимать архитектуру по самому проекту. Логика — в редакторе AppSheet (видно), не в скрытом Apps Script.

## §4. Все формулы дословно

### §4.1. Дашборд — Живая зона (открытый период)

**`settings[dash_открытый_поступления]`** (Number, VC):
```
SUM(SELECT(payments[amount],
  AND(
    YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
    MONTH([payment_date]) = NUMBER(RIGHT(ANY(settings[current_period]), 2))
  )
))
```

**`settings[dash_открытый_расходы]`** (Number, VC):
```
SUM(SELECT(expenses[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**`settings[dash_открытый_начислено]`** (Number, VC):
```
SUM(SELECT(charges[amount],
  NUMBER([period]) = NUMBER(ANY(settings[current_period]))
))
```

**Архитектурное наблюдение:** три формулы используют **разные** способы определить «текущий период»:
- Поступления — по `payments[payment_date]` через `YEAR()`+`MONTH()`. Потому что у `payments` нет `period`.
- Расходы и начисления — по `[period] = current_period`. Потому что у `expenses` и `charges` `period` есть.

Это **намеренно**, не баг. Следствие: платёж бэк-датой за март попадает в мартовские поступления, не в апрельские.

### §4.2. Дашборд — Долги и Твёрдая земля

**`settings[dash_долги_количество]`** (Number, VC):
```
COUNT(SELECT(families[family_id], [family_overdue_debt] > 0))
```

**`settings[dash_долги_сумма]`** (Number, VC):
```
SUM(SELECT(families[family_overdue_debt], [family_overdue_debt] > 0))
```

**`settings[dash_земля_получено]`** (Number, VC):
```
SUM(SELECT(payments[amount],
  AND(
    [payment_date] >= ANY(settings[academic_year_start]),
    OR(
      YEAR([payment_date]) < NUMBER(LEFT(ANY(settings[current_period]), 4)),
      AND(
        YEAR([payment_date]) = NUMBER(LEFT(ANY(settings[current_period]), 4)),
        MONTH([payment_date]) < NUMBER(RIGHT(ANY(settings[current_period]), 2))
      )
    )
  )
))
```

**`settings[dash_земля_израсходовано]`** (Number, VC):
```
SUM(SELECT(expenses[amount],
  AND(
    NUMBER([period]) >= YEAR(ANY(settings[academic_year_start])) * 100 + MONTH(ANY(settings[academic_year_start])),
    NUMBER([period]) < NUMBER(ANY(settings[current_period]))
  )
))
```

**`settings[dash_земля_свободные]`** (Number, VC):
```
[dash_земля_получено] - [dash_земля_израсходовано]
```

**`settings[dash_пред_период]`** (Number, VC) — хелпер:
```
IFS(
  NUMBER(RIGHT(ANY(settings[current_period]), 2)) > 1,
  NUMBER(LEFT(ANY(settings[current_period]), 4)) * 100 + NUMBER(RIGHT(ANY(settings[current_period]), 2)) - 1,
  TRUE,
  (NUMBER(LEFT(ANY(settings[current_period]), 4)) - 1) * 100 + 12
)
```

Возвращает YYYYMM предыдущего месяца с корректным переходом через границу года.

### §4.3. Текущая задолженность

**`students[debt]`** (Number, VC) — баланс сосуда ученика:
```
SUM(SELECT(charges[amount], [student_id] = [_THISROW].[student_id]))
- SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
- SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id]))
```

Может быть положительным, нулевым, отрицательным.

**`families[family_debt]`** (Number, VC):
```
SUM(SELECT(students[debt], [family_id] = [_THISROW].[family_id]))
```

**Без обрезки нулём** — переплата одного ученика **компенсирует** долг другого внутри семьи.

### §4.4. Дебиторская задолженность (просрочка)

**`students[overdue_debt]`** (Number, VC) — только просроченная часть:
```
IFS(
  SUM(SELECT(charges[amount],
    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))
    )
  ))
  - SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id]))
  > 0,
  SUM(SELECT(charges[amount],
    AND(
      [student_id] = [_THISROW].[student_id],
      NUMBER([period]) < NUMBER(ANY(settings[current_period]))
    )
  ))
  - SUM(SELECT(payments[amount], [student_id] = [_THISROW].[student_id]))
  - SUM(SELECT(opening_balances[amount], [student_id] = [_THISROW].[student_id])),
  TRUE,
  0
)
```

Эквивалент `MAX(0, ...)`, написанный длинно через IFS. Выражение продублировано — **при правке менять в обоих местах**. Кандидат на упрощение через `MAX(0, ...)` в будущем.

**`families[family_overdue_debt]`** (Number, VC):
```
SUM(SELECT(students[overdue_debt], [family_id] = [_THISROW].[family_id]))
```

**Каждый `overdue_debt` уже обрезан нулём** — переплата одного ученика **не** компенсирует просрочку другого.

**Намеренная асимметрия с `family_debt`** (подтверждено владельцем 03.05.2026): для текущего баланса семья = единый сосуд, для просрочки — каждый ученик отдельно.

CONCATENATE([first_name], " ", [last_name], " (", [class_group], ")")

### §4.6. current_period

**`settings[current_period]`** (Ref → ref_periods, VC):
```
INDEX(SORT(SELECT(charges[period], [tariff_id].[tariff_type] = "Регулярный"), TRUE), 1)
```

⚠️ **На 04.05.2026 формула содержит фильтр `tariff_type = "Регулярный"`, но в реальности значения `tariff_type` у всех тарифов — `"Авто"`.** Это рудимент. Формула фактически работает (фильтр `= "Регулярный"` не матчит ничего, но потом TRUE-ветка SORT/INDEX всё равно даёт результат), однако это надо проверить и при необходимости поправить.

⚠️ Вообще — **всё разделение `tariff_type` в `tariffs` — рудимент** (см. §6.5). Если убирать `tariff_type` из системы, эту формулу надо переписать без фильтра, типа:
```
INDEX(SORT(charges[period], TRUE), 1)
```

### §4.7. opening_balances в формулах долга

Обе формулы (§4.3, §4.4) **вычитают** `SUM(opening_balances[amount])` для ученика. Это фундамент модели сосудов на момент запуска биллинга. Если таблица пустая — вычитается ноль.

### §4.8. payments без фильтра по дате

В формулах `debt` и `overdue_debt` платежи учитываются **все**, без фильтра по дате. Это правильно по модели сосудов: деньги падают в общий сосуд, без привязки к месяцу.

## §5. Архитектура #7 (закрытие периода) — на 04.05.2026 в текущем состоянии

### §5.1. Цель

Кнопка «Закрыть период» в фрейме «скрипты» (MENU NAVIGATION). По клику: подтверждение → массовая генерация регулярных начислений на следующий месяц для активных учеников → сообщение о результате. Защита от повторного нажатия в тот же месяц.

### §5.2. Финальная архитектура

**Платформа:** AppSheet Bot, триггерится через служебную колонку `settings[last_close_trigger]`.

**Почему через служебную колонку:** в AppSheet нет ручного триггера для Bot (подтверждено по доке `support.google.com/appsheet/answer/...`, см. ниже). Идиоматичный паттерн: action обновляет служебное поле → Data Change Event → Bot.

**Цепочка:**

1. Пользователь на view «скрипты» нажимает кнопку «Закрыть период».
2. Срабатывает Action `settings__close_period_trigger` (`Data: execute an action on a set of rows`, Referenced Table=`settings`, Referenced Action=`settings__update_trigger`).
3. `settings__update_trigger` пишет `NOW()` в `settings[last_close_trigger]`.
4. Bot `bot_close_period` ловит изменение через Event с условием `[_THISROW_BEFORE].[last_close_trigger] <> [_THISROW_AFTER].[last_close_trigger]`.
5. Bot выполняет Branch:
   - **YES (период уже закрыт)** → уведомление «Период уже закрыт».
   - **NO** → ForEach по `student_tariffs` (фильтр: активные + регулярный тариф) → Action `student_tariffs__create_next_charge` создаёт `charges` для каждого → финальное уведомление «Создано N, должников T».

### §5.3. Action и колонки — текущее состояние

**`settings[last_close_trigger]`** — DateTime, Show=false, физическая в Sheets и AppSheet. Создана.

**Action `settings__update_trigger`:**
- For a record of this table: `settings`
- Do this: `Data: set the values of some columns in this row`
- Set columns: `last_close_trigger` = `NOW()`
- Show_If: проверить (на момент закрытия дня была временная диагностика, могла остаться `TRUE` или `FALSE` — TODO для #10 сверить).

**Action `settings__close_period_trigger`:**
- For a record of this table: `settings`
- Do this: `Data: execute an action on a set of rows`
- Referenced Table: `settings`
- Referenced Rows: `SELECT(settings[id], TRUE)` или эквивалент (точное имя ключа — TODO сверить, в §3.10 я не зафиксировал)
- Referenced Action: `settings__update_trigger`
- Display name: «Закрыть период»
- Show_If: `AND(LOOKUP(USEREMAIL(), "users", "email", "role") = "ADMIN", CONTEXT("View") = "скрипты")` — кнопка только в фрейме «скрипты», только для ADMIN.
- Needs confirmation: ON
- Confirmation Message: написана через `CONCATENATE` (синтаксис `<<...>>` в Confirmation Message **не работает** в AppSheet). Текст: «Вы точно хотите закрыть период YYYYMM? Следующий открытый период будет YYYY(MM+1).» — где YYYYMM = `current_period`, YYYY(MM+1) — следующий из `ref_periods`.

**Action `student_tariffs__create_next_charge`:**
- For a record of this table: `student_tariffs`
- Do this: `Data: add a new row to another table using values from this row`
- Table to add to: `charges`
- Set columns:
  - `charge_id` = `UNIQUEID()`
  - `student_id` = `[student_id]`
  - `tariff_id` = `[tariff_id]`
  - `charge_type` = `"Авто"` (⚠️ **в брифинге было `"Регулярный"` — это была ошибка стратега, в реальности значение `"Авто"`**)
  - `period` = `INDEX(SORT(SELECT(ref_periods[period], NUMBER([period]) > NUMBER(ANY(settings[current_period])))), 1)`
  - `amount` = `[tariff_id].[amount]`
  - `description` = `""`
  - `created_at` — не маппится (Initial Value = `NOW()`)
- Show_If: `FALSE`.

**Action `tariffs__close_period_trigger`** — старая версия из ранней архитектуры. **Должна быть скрыта** (Show_If=FALSE) или удалена. На 04.05.2026 точное состояние — TODO сверить.

## §7. Открытые «руки» — карта подводных камней AppSheet

Записаны ситуации, в которых Sonnet поднимал руки в #7 — для будущих брифингов:

6. **`Same-table reference action`** разрешён (паттерн loop в AppSheet). Если цепочка не работает — причина не в same-table, а в чём-то другом (Show_If = FALSE у Referenced Action может блокировать вызов).
