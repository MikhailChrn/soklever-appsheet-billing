# Отчёт SubGeneral #15 — фикс рассинхрона `current_period`

**Дата:** 13.05.2026
**Версия AppSheet на входе:** 1.000569 (после v14)
**Версия AppSheet на выходе:** 1.000573
**Исполнитель:** SubGeneral в Claude Code + владелец (руками в AppSheet)
**Тип:** диагностическая сессия + горячий фикс

---

## §1. Краткая суть

Найден и устранён баг закрытия периода, обнаруженный в конце #14 (см. v14 §3.0). Бот `bot_close_period` создавал charges не за следующий период (202606), а за прошлый (202604), при том что плитка дашборда показывала `current_period = 202605`.

**Корень:** виртуальная колонка `current_period` внутри bot transaction возвращала устаревшее значение (snapshot, не пересчитанный). Frontend и Data Explorer видели свежее значение, бот — старое.

**Решение:** перевод `current_period` из VC в физическую колонку, обновление перед запуском бота через action `settings__update_trigger`.

**Результат тестового прогона:** бот создал 8 charges за 202606 (для всех 8 активных учеников на T010). Сообщение: «Период 202606 закрыт. Открытый период: 202607. Создано начислений: 8.» — корректно.

---

## §2. Диагностика (хронология)

### §2.1. Симптом

После летнего перевода `bot_summer_apply` (12.05.2026) запуск `bot_close_period`:
- В Sheets создались 8 строк charges за **202604** (вместо ожидаемых 202606).
- Сообщение бота: «Период 202603 закрыт. Открытый период: 202604. Создано начислений: 5.»
- Плитка дашборда: «Открытый период: 202605» — правильно.
- Confirmation перед запуском: «Следующий открытый период будет 202606» — правильно.

Воспроизводилось стабильно: при повторных нажатиях создавались **новые** 8 строк за 202604.

### §2.2. Поэтапная проверка

1. **Формула VC `current_period`** = `INDEX(SORT(SELECT(charges[period], TRUE), TRUE), 1)`. Через Data Explorer — даёт 202605. ✓
2. **Формула `period` в action `student_tariffs__create_next_charge`** = `INDEX(SORT(SELECT(ref_periods[period], NUMBER([period]) > NUMBER(ANY(settings[current_period])))), 1)`. Через Data Explorer — даёт 202606. ✓
3. **Branch condition** в `step_branch_check` — даёт N (NO), корректно идёт в создание charges. ✓
4. **Monitor реального запуска бота:** в Output Input-шага лежит `current_period: 202605`, но `dash_пред_период: 202602` (должен быть 202604 — формула `current_period - 1`). Расхождение **внутри одного snapshot**.
5. **Замена `ANY(settings[current_period])` на `INDEX(SORT(SELECT(charges[period], TRUE), TRUE), 1)`** в формуле сообщения `last_event_message` — **не помогло**. Бот всё равно записал 202603/202604.

### §2.3. Запрос к Gemini

Gemini подтвердил гипотезу: AppSheet кеширует snapshot таблиц при старте bot transaction. VC, `ANY()`, `SELECT()` из другой таблицы внутри бота читают этот кеш, а не свежие данные. Главная рекомендация — перевод VC в физическую колонку.

### §2.4. Дополнительная аномалия

Сообщение «Создано: 5» при 8 активных учениках. SELECT в step_no_create_charges через Data Explorer возвращает 8 assignment_id. То есть бот видит и таблицу `student_tariffs` тоже в устаревшем состоянии (5 строк — до `bot_summer_apply`).

Это **закрепило диагноз**: внутри bot transaction замораживается весь snapshot всех таблиц.

## §3. Что сделано (по шагам)

### §3.1. Подготовка

1. В Sheets, лист `settings`, добавлена физическая колонка `current_period` (столбец N), вручную заполнена значением `202605` (текущее реальное).
2. В AppSheet — удалена VC `current_period` (Data → settings → корзина). Возникло 18+ ошибок «Unable to find column CURRENT_PERIOD» в зависимых формулах (дашборд, valid_if на charges/expenses, formula в action `student_tariffs__create_next_charge` и др.).
3. **Regenerate Schema** на таблице settings — подхватил физическую колонку из Sheets.
4. Тип колонки `current_period` после Regenerate — `Duration` (AppSheet угадал неправильно). Изменён на **Ref → ref_periods** (как было у VC).
5. Show? — снят (системная колонка, в формах не нужна).
6. SAVE — все 18+ ошибок ушли. Формулы автоматически переключились на физическую колонку.
7. Sync на клиенте — дашборд показывает «Открытый период: 202605». ✓

### §3.2. Расширение `settings__update_trigger`

Action ранее писал только `last_close_trigger = NOW()`. Добавлен второй Set:
- `current_period = INDEX(SORT(SELECT(charges[period], TRUE), TRUE), 1)`

Семантика: перед запуском бота закрытия в физическую колонку записывается актуальный максимум `charges[period]`. Bot transaction стартует уже с правильным значением, замороженный snapshot теперь содержит свежие данные.

`bot_summer_apply` (action `settings__set_summer_trigger`) **не трогали** — он не меняет periods в charges, значит `current_period` после него не должен меняться.

### §3.3. Откат тестовых правок

Формула `last_event_message` в `step_no_set_result` была временно расширена на прямое чтение charges без посредника settings (для проверки гипотезы, что VC settings — единственная причина). Гипотеза не подтвердилась — расширенная формула давала тот же баг. **Решено оставить расширенную версию** как более устойчивую: меньше зависимостей. Откат не требуется.

### §3.4. Очистка мусора

В Sheets удалены 16 строк-мусора:
- 8 строк за 202604 (created_at 12.05.2026 20:35:23) — от багового прогона #14.
- 8 строк за 202604 (created_at 13.05.2026 13:04:20 и 16:19:18) — от двух багующих прогонов в ходе диагностики.

После очистки в charges осталось: 5 строк за 202603, 5 за 202604, 6 за 202605 — итого 16 строк (исходное состояние плюс майские из прогона до бага).

### §3.5. Финальный тестовый прогон

1. Sync в AppSheet.
2. Нажатие «закрыть период».
3. Confirmation: «Вы точно хотите закрыть период 202605? Следующий открытый период будет 202606.» ✓
4. После выполнения: в Sheets создались **8 строк** за **202606** на T010 (для всех 8 активных учеников).
5. Сообщение бота: «Период 202606 закрыт. Открытый период: 202607. Создано начислений: 8.»

**Артефакт сообщения:** говорит «Период 202606 закрыт» вместо ожидаемого «202605». Это потому, что `current_period` теперь физическая, и `step_no_set_result` строит сообщение через `ANY(settings[current_period])` **после** ForEach. На момент построения сообщения action `settings__update_trigger` уже записал в `current_period` свежий max (202606, после создания новых charges). По существу действие выполнено правильно: charges за 202606 созданы, сообщение косметически опережает. Принято как мини-хвост — см. §5.

## §5. Хвосты

### §5.1. Косметический баг в сообщении бота

«Период 202606 закрыт» вместо «Период 202605 закрыт». Причина — `settings__update_trigger` обновляет `current_period` **до** ForEach (нужно для action `create_next_charge`), а сообщение в `step_no_set_result` строится **после** ForEach, когда max(charges[period]) уже сместился.

**Возможный фикс:** хранить «закрытый период» отдельно (например, передавать в Set Values константой, или вычислять как `current_period - 1` через IFS-формулу разворота YYYYMM). Не блокер — действие выполнено правильно, страдает только текст.

### §5.2. Поведение действия `student_tariffs__create_next_charge`

Формула `period` в action осталась через `ANY(settings[current_period])`. После перехода на физику работает корректно. **Менять не нужно.** Но в SKILL зафиксировать паттерн: формулы в action, которые читают settings, работают через физические колонки, не через VC.

### §5.3. Учебный материал для SKILL

Полученный паттерн — фундаментальный для AppSheet. Должен быть зафиксирован в SKILL как принцип: **«если значение нужно прочитать внутри bot transaction — оно должно быть физической колонкой, обновляемой триггер-action'ом перед запуском бота. VC внутри бота читаются из замороженного snapshot и могут отставать.»**
