## §2. Базовые принципы

### §2.1. Модель сосудов

Деньги в системе — это уровни в сосудах, а не цепочка транзакций. У каждого ученика виртуальный «сосуд». Платёж поднимает уровень, начисление опускает. Долг = отрицательный уровень. Нет привязки платежа к конкретному начислению, нет «закрытых» начислений.

### §2.2. Живая зона vs твёрдая земля

- **Живая зона** = открытый период (`settings[current_period]`). Зона, где данные ещё могут двигаться.
- **Твёрдая земля** = всё, что строго раньше `current_period`. Закрытые периоды, зафиксированная история.

### §2.3. Суровая правда без шума

Дашборды и отчёты показывают факты. Не прогнозы, не интерпретации, не предсказания. Руководитель сам делает выводы из цифр.

### §2.4. Простота и лаконичность

Система ориентирована на сверх-малый бизнес. Не усложнять модель ради «потенциальных» сценариев. Лучше потерять историю, чем содержать неиспользуемые поля.

## §3. Архитектурные решения (Records)

### §Р-1. У `payments` нет поля `period`

Платёж имеет только `payment_date`. Привязка к периоду не делается — деньги попадают в общий сосуд ученика.

### §Р-2. У `expenses` есть поле `period`

Расход относится к календарному месяцу через явное `period` (YYYYMM). Дополнительно есть `expense_date` (когда фактически проведён). Для дашборда и аналитики **используется `period`**, не `expense_date`.

### §Р-5. Динамический долг без истории

Долг ученика/семьи рассчитывается налету через VC. Таблица `debts` в системе **не используется** (см. О-2).

### §Р-6. Должник = `family_overdue_debt > 0` по периодам строго раньше активного

`overdue_debt` ученика = сумма начислений минус сумма платежей по периодам строго раньше `current_period`, обрезанная снизу нулём. `family_overdue_debt` = сумма `overdue_debt` всех учеников семьи. View «Должники» показывает семьи с `family_overdue_debt > 0`.

### §Р-7. Закрытие периода — в конце месяца

Регламентно — 28–31 числа. Запускается **вручную кнопкой**, не по расписанию. Точная процедура — см. О-1 (решён).

### §Р-8. Тарифы и инфляция — без истории

При изменении `tariffs[amount]` сумма переписывается в той же строке. История изменений не сохраняется. Старые `charges` не пересчитываются — в них `amount` зафиксирован на момент создания.

**Дополнение от 01.05.2026:** колонки `valid_from`, `valid_to` в `tariffs` функционально не использовались и **удалены** в брифинге #4. Структура `tariffs`: `tariff_id | tariff_name | tariff_type | amount | period | comment`.

### §Р-9. Дашборд «Главная» — реализован

**Статус:** ✅ закрыт в брифинге #3 (30.04–01.05.2026).

**Архитектура — три блока сверху вниз:**

**Блок 1. Живая зона (открытый период):**
1. Поступления за открытый период (`dash_открытый_поступления`).
2. Расходы за открытый период (`dash_открытый_расходы`).
3. Всего начислено в открытом периоде (`dash_открытый_начислено`).

**Блок 2. Долги (по закрытым периодам, по Р-6):**
1. Количество должников-семей (`dash_долги_количество`).
2. Общий долг (`dash_долги_сумма`).

**Блок 3. Твёрдая земля (с начала учебного года):**
1. Получено в закрытых периодах (`dash_земля_получено`).
2. Израсходовано в закрытых периодах (`dash_земля_израсходовано`).
3. Свободные средства (`dash_земля_свободные`).

**Реализация:** все 9 показателей — VC в `settings`. Три slice (`settings_dash_открытый`, `settings_dash_долги`, `settings_dash_земля`). Три Detail view + один Dashboard view «Главная».

**Доступ:** только ADMIN. OPERATOR не видит. Реализовано через `Show if = LOOKUP(USEREMAIL(), "users", "email", "role") = "ADMIN"`.

**Заголовки sub-view — через формулу в Display name view** (новый паттерн, см. BP).

**Имена колонок:** все начинаются с `dash_открытый_*`, `dash_долги_*`, `dash_земля_*`. Был принят более понятный префикс «открытый», а не «живая» (отклонение от первоначального брифинга).

### §Р-11. Массовая смена тарифа на лето

В конце учебного года скрипт массово меняет тариф у всех учеников на «летний амортизационный». С 1 сентября — ревизия учеников, смена классов и присвоение новых регулярных тарифов вручную.

**Дедлайн на реализацию скрипта: 1 июня 2026** (см. О-4 и брифинг #8 в §7).

### §Р-12. Цветовая индикация в UI

Используется в дашборде и view «Должники» для подсветки нулей, отрицательных значений, активного периода. Конкретные цвета — на усмотрение Sonnet, согласовываются по ходу.

### §Р-14. `student_tariffs` без дат, без истории

**Принят:** 01.05.2026, реализован в брифинге #4.

**Структура:** `assignment_id | student_id | tariff_id | comment` (поля `date_from`, `date_to` / `assigned_from`, `assigned_to` удалены).

**Логика:**
- У ученика **0–N одновременных тарифов**. Типичный кейс: «Основной» + «Продлёнка».
- Скрипт закрытия (#7) в момент запуска читает текущий `student_tariffs` ученика и генерит по одному `charges` на каждую строку.
- Смена тарифа = перезапись `tariff_id` или удаление/добавление строки. Старые `charges` не трогаются (Р-8).
- Удаление строки = со следующего закрытия начисления по этому тарифу не создаются.

**Обоснование:** §2.4 — простота. История косвенно восстанавливается из `charges`.

**Форма редактирования:** реализована в брифинге #4 как inline-блок «Активные тарифы» в карточке ученика (Deck-view с действиями Add/Edit/Delete). Дополнительно — блок «связанные ученики» в карточке тарифа (Table-view через slice).

## §4. Решения по сущностям

### §4.1. Семья — единица долга

Долг считается и отображается **по семье**, не по ученику. View «Должники» работает с `families[family_overdue_debt]`. Нотификации (если будут) — на семью, не на ученика.

### §4.2. Регулярные vs разовые начисления

`charges[charge_type]` — Enum: «Регулярный», «Разовый». Скрипт закрытия генерит **только регулярные**. Разовые (кружки, доп. услуги, разовые мероприятия) — заводятся вручную через форму.

### §4.3. Кто видит что

| Роль | Доступ |
|---|---|
| ADMIN (владелец) | Всё |
| OPERATOR | Только форма расхода + свои расходы (по `created_by = USEREMAIL()`) |

Роль определяется через `LOOKUP(USEREMAIL(), "users", "email", "role")`. На каждый view, скрытый от OPERATOR — `Show if` с условием. На slice `expenses_filtered` — ролевой фильтр в Row filter (см. ниже).

**Шаблон ролевого фильтра в slice:**
```
OR(
  LOOKUP(USEREMAIL(), "users", "email", "role") = "ADMIN",
  [created_by] = USEREMAIL()
)
```
Если регистр email не совпадает — обернуть в LOWER().

## §5. Best Practices (BP)

### Базовые BP (из v6)

- **BP-1.** Sheets — источник истины для владельца. AppSheet — UI для оператора. Любое массовое изменение делается в Sheets, единичные правки — через AppSheet.
- **BP-17.** Если формула в slice не сохраняется — пересоздать slice.
- **BP-18.** Slice по умолчанию `Update mode = ADDS_AND_UPDATES`, если не оговорено иное.
- **BP-19.** Заголовок формы — отдельная VC типа Show с `Category = Section_Header`, App formula `""`. Помещается первой в Manual order формы.
- **BP-23.** Show? для заголовочной VC формы = `true` (без условий, иначе AppSheet может скрыть заголовок).
- **BP-28.** При создании slice или правке Slice Columns — сначала ставить `*`, после Regenerate Schema указывать явный список. Иначе AppSheet может не подхватить новые VC.
- **BP-30.** KEY-колонка скрывается через Show? в форме, но **не может быть исключена из Slice Columns** (AppSheet требует ключа в slice).

### Новые BP (из v7, отчёты Sonnet по #3 и #4)

**BP-29. Display name view = динамическая формула.**
Технические имена view (`dash_открытый_detail` и т.п.) выглядят непрофессионально как заголовки блоков в Dashboard. В настройках view → Display → Display name можно задать AppSheet-формулу. Контекста строки нет, поэтому через `ANY(table[column])`. Пример:
```
CONCATENATE("Открытый период: ", ANY(settings[current_period]))
```
Этот паттерн заменяет необходимость в заголовочных VC типа Show внутри slice.

**BP-30 (расширенный). `academic_year_start` хранить в Sheets, не вычислять.**
AppSheet при Regenerate Schema может подставить автоформулу `IF(MONTH(TODAY())...)` на Date-колонку без формулы. Это перезаписывает значение из Sheets. После Regenerate проверять колонку — App formula должна быть **пустой**.

**BP-31. Ref-колонка скрыта → не появляется в форме.**
Если Ref-колонка не отображается в списке Column order формы (Manual → + Add) — причина в Show? = false. Включить Show? = true перед настройкой формы.

**BP-32. VC типа Show не вычисляется для новых строк в форме.**
VC типа Show с формулой на основе Ref не показывает значение при добавлении новой записи (нет контекста строки). Использовать тип Text — вычисляется корректно. Технически поле останется input, но App formula перезаписывает ввод при сохранении, и визуально оно выглядит как read-only текст серым.

**BP-33. Referenced table name принимает только таблицы и слайсы.**
В поле Referenced table name (Type Details → Related X) отображаются только таблицы и слайсы — не views. Для кастомного отображения Related-блока создавать slice и указывать его.

**BP-34. Конфликт лишнего Table view и inline Deck.**
Симптом: Related-блок в карточке отображается как таблица с чужими колонками вместо deck. Причина: существующий ref-position Table view на дочерней таблице перехватывает рендеринг Related-блока. Решение: удалить лишний Table view. Если нужен для другого контекста — использовать slice + явное указание Referenced table name. После удаления — перезапустить приложение.

**BP-35. KEY скрывается через Show? = false на самой колонке.**
KEY-колонку нельзя исключить из Slice Columns (AppSheet требует), но можно скрыть от рендеринга через `Show? = false` на самой колонке таблицы. Возможно появление двух warnings: «unsearchable» (информационный, игнорировать) и «label may not be hidden» (если LABEL стоял на этой колонке — переставить LABEL на читаемое поле, например `_student_name`).

3. Собирает реестр всех вхождений с указанием места (таблица, колонка, slice, view, формула).
4. Возвращает реестр SubGeneral.

**Формат реестра:** таблица с колонками «Где / Что / Текущее значение / Тип использования / Зависимости».
