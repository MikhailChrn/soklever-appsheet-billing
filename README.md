# СО Клевер Биллинг — База знаний

Долгоживущая память проекта **СО Клевер Биллинг** — системы биллинга для частной школы на стеке Google Sheets + AppSheet (Этап 0, «Google-прародитель»).

**Дедлайн запуска:** 01.06.2026.

---

## Где я и куда смотреть

### Если ты SubGeneral новой ветки

1. **Прочитай `SYSTEM.md`** целиком — бизнес-модель и контекст.
2. **Прочитай `GLOSSARY.md`** — рабочие метафоры и термины.
3. **Прочитай последний бэкап** в `subgeneral-backups/` — текущее рабочее состояние и хвосты.
4. **Спроси у владельца:**
   - Сегодняшнюю дату.
   - Текущую версию AppSheet.
   - Что в приоритете на день.
   - Изменения после последнего бэкапа.

`ARCHITECTURE.md` и `RECOVERY.md` — читать по необходимости (когда нужны технические детали или процедура восстановления).

### Если ты владелец и забыл, где что

- **Что система делает и зачем** → `SYSTEM.md`.
- **Что значит «дебиторка», «живая зона», «сосуды»** → `GLOSSARY.md`.
- **Как реализована конкретная фича в AppSheet** → `ARCHITECTURE.md` (после сборки).
- **Как делать бэкап, формат брифинга, регламенты** → `REGULATIONS.md` (после сборки).
- **Как восстановить систему с нуля** → `RECOVERY.md` (после сборки в #09).
- **Состояние проекта на конкретную дату** → `subgeneral-backups/<дата>_v<N>.md`.
- **Конкретный брифинг для Sonnet** → `briefings/`.
- **Конкретный отчёт от Sonnet** → `reports/`.

### Если ты Sonnet

Ты не работаешь с этим репозиторием напрямую. Тебе приходит брифинг (`.md`-файл) с контекстом задачи и ссылками на нужные разделы базы знаний. Если в брифинге не хватает деталей — поднимаешь руку через владельца к стратегу (SubGeneral).

## Принципы работы с базой

1. **Корневые `*.md`** редактируются только при изменении системы или решений. Не при каждом брифинге.
2. **`subgeneral-backups/` пишется при закрытии каждой ветки** SubGeneral. Старые не удаляются — это история.
3. **`briefings/` и `reports/` — append-only.** Не редактируются после создания.
4. **`README.md`** обновляется при крупных вехах (запуск, переход на новый этап).
