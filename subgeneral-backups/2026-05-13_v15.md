## §1. Что произошло за #15 (13.05.2026)

Однодневная ветка. **Главное событие — устранён баг закрытия периода**, обнаруженный в конце #14 (см. v14 §3.0). Параллельно — старт работы над брифом pre-#09 (взаимозачёты).

### §1.1. Диагностика рассинхрона `current_period`

Симптом: `bot_close_period` создавал charges за 202604 вместо 202606. Плитка дашборда показывала 202605, Confirmation показывал 202606, бот создавал 202604.

Поэтапная проверка показала: формулы корректны во всех местах (VC `current_period`, action `create_next_charge`, Branch condition, message). Data Explorer для каждой даёт правильный ответ. **Но внутри bot transaction результат другой.**

Запрос к Gemini подтвердил: AppSheet **замораживает snapshot всех таблиц** при старте bot transaction. VC, `ANY()`, `SELECT()` внутри бота читают этот кеш, а не свежие данные. Если VC зависит от `charges`, а snapshot замёрз в момент `bot_summer_apply` — VC даёт значение, актуальное для того момента.

Дополнительное подтверждение в Monitor: Output Input-шага содержал одновременно `current_period: 202605` и `dash_пред_период: 202602` — внутри одного snapshot разные VC дают несогласованные значения (разные моменты заморозки).

### §1.2. Решение

Перевод `current_period` из VC в физическую колонку. Action `settings__update_trigger` расширен: вместе с `last_close_trigger = NOW()` пишет `current_period = INDEX(SORT(SELECT(charges[period], TRUE), TRUE), 1)`. Перед запуском бота физическая колонка обновляется свежим max, и bot transaction стартует с актуальным значением.

Все 18+ зависимых формул (дашборд, valid_if, action `create_next_charge`, сообщение бота) **автоматически переключились** на физическую колонку после переименования. Никакая семантика не сломалась.

### §1.3. Тестовый прогон

Очищены 16 строк-мусора в charges (от трёх багующих прогонов 12-13.05). Запущено «закрыть период» — создались 8 charges за 202606 (все 8 активных учеников, T010). Сообщение: «Период 202606 закрыт. Открытый период: 202607. Создано: 8.» — корректно по существу, косметически опережает (см. §3.1).

### §1.4. Подготовка брифа pre-#09

Бриф пишется в `briefings/2026-05-13_briefing-pre09.md`. Тема — взаимозачёты: при сохранении `payment` с `payment_type = "Взаимозачёт"` автоматический LINKTOFORM на форму expense с предзаполнением. См. v14 §3.3 и SYSTEM.md §8.5.

## §3. Открытые хвосты (наследуются в #16)

### §3.1. Косметический баг в сообщении бота

Сообщение «Период 202606 закрыт» вместо «Период 202605 закрыт». Причина — `settings__update_trigger` обновляет `current_period` до ForEach, сообщение строится после. Действие выполнено правильно, страдает только текст.

**Возможный фикс:** хранить «закрытый период» отдельно (вычислять `current_period - 1` в формуле сообщения через тот же IFS-разворот YYYYMM). Не блокер, но желательно поправить до боевого мая, чтобы пользователь видел правильную фразу.

### §3.2. Перенос фикса на боевой контур

Сейчас фикс на тестовой схеме. Перед боевым закрытием мая (28-31.05) — повторить шаги на копии прода:

3. Удалить VC, Regenerate Schema, поставить тип Ref, снять Show.

### §3.3. Перенос фикса для `bot_summer_apply`

`bot_summer_apply` сейчас НЕ обновляет `current_period`. Аргумент: он не меняет periods в charges. Это верно для текущей реализации. **Если** в будущем летний перевод начнёт создавать charges (например, июньские сразу при переключении) — нужно будет добавить обновление `current_period` в action `settings__set_summer_trigger`.

**Принципиальное архитектурное знание, закреплённое сегодня:**

> Внутри AppSheet bot transaction VC и `SELECT()` из других таблиц читают **замороженный snapshot**, не пересчитываются. Если значение должно быть актуальным в момент работы бота — оно должно быть **физической колонкой**, обновляемой триггер-action'ом перед запуском бота.

Это касается не только `current_period`. В будущих архитектурных решениях по любому новому боту — спрашивать «какие VC он читает из снапшота, нужно ли их перевести в физику».
